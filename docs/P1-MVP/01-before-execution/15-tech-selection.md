# 技术选型文档

## 1. 本轮技术目标

为“书童四九”本地文件管理与发布服务确定可直接进入架构阶段的技术主选项，明确底座、业务壳、对外访问层和局部参考对象，并排除不适合本轮主路线的技术方向。

## 2. 已确认的输入约束

- 上游输入为 `10-requirements/business_rules_memo.md`，状态为 `approved`。
- 系统部署环境为 `1Panel + MiniPC` 单机自托管场景。
- 系统必须统一承接文件与 HTML 富文本两类内容。
- 系统必须向本地智能体提供 Open API 写入能力。
- 系统必须支持内容列表、搜索、详情、分享访问和基础权限隔离。
- 当前优先级是先打通“本地写入 -> 对外访问”的主链路，再逐步增强分享治理和生命周期能力。

## 3. 技术选择结论

### 3.1 主路线选择

本轮主路线选择：`轻量后端底座 + 自研薄业务壳 + 自研对外访问层`。

### 3.2 底座选择

本轮底座选择：`PocketBase`。

选择理由：

- 作为单机自托管底座，复杂度明显低于 `Appwrite` 和 `Supabase`。
- 同时具备文件存储、数据存储、用户管理和 API 能力，能承接本项目的基础数据面。
- 更适合作为“书童四九”大项目中的本地子模块，而不是独立的重平台。
- 能为后续业务壳补充内容归属、分享访问和 hash URL 规则留下清晰边界。

### 3.3 业务壳选择

本轮业务壳选择：`自研一层较薄的业务服务`。

该层负责：

- API Key 到业务用户的映射。
- 文件与富文本的统一内容模型。
- 内容归属与权限规则。
- 分享链接生成、撤销与访问校验。
- hash URL 的生成与解析。
- 面向前台的列表、搜索、详情 API。

### 3.4 对外访问层选择

本轮对外访问层选择：`按本项目需求自研网页层`，不直接采用现成建站系统或文件分享系统作为主前台。

选择理由：

- 本项目的对外访问不是通用站点建设问题，而是内容发布与只读访问问题。
- 前台需要围绕统一内容模型和分享规则定制，而不是围绕文章、站点或纯文件树模型展开。

## 4. 候选方案比较后的取舍结果

### 4.1 采用：轻量后端 / BaaS 路线

采用原因：

- 最贴近“API-first、本地子系统、统一内容底座”这三个核心约束。
- 能在不引入重平台的前提下承接文件、元数据和基础用户能力。

### 4.2 不采用：Headless CMS / 内容平台作为主承载系统

不采用原因：

- 更偏内容运营平台，平台边界偏重。
- 适合作为内容建模参考，不适合作为本轮底座主路线。

### 4.3 不采用：文件分享产品作为主承载系统

不采用原因：

- 分享能力强，但统一内容模型、HTML 富文本承接和内容检索能力不足。
- 更适合作为分享治理和上传体验的参考对象。

### 4.4 不采用：建站 / 笔记系统作为主承载系统

不采用原因：

- 系统中心偏站点或笔记，不适合作为智能体驱动写入的内容底座。
- 会把运行时和数据模型带向错误方向。

## 5. 底座候选的最终选择理由

### 5.1 为什么不选 Appwrite

- 平台能力完整，但部署与运行形态更重。
- 对当前 MiniPC 单机场景属于平台边界大于问题边界。

### 5.2 为什么不选 Supabase

- 数据与平台能力很强，但组件栈更复杂。
- 更适合以数据库平台为中心的系统，而不是一个本地轻量内容服务子模块。

### 5.3 为什么选择 PocketBase

- 轻量、单机友好、适合本地自托管。
- 同时覆盖文件、记录、用户和 API 的基础需求。
- 更适合作为“底座 + 薄业务壳”模式中的底层核心。

## 6. 可复用开源仓库 / 公开实现检查结果

### 6.1 主底座

- `PocketBase`：作为本轮唯一主底座选择。

### 6.2 局部能力参考对象

- `Gokapi`：参考分享链接生命周期、过期和访问限制策略。
- `chibisafe`：参考 API Key 上传接入、上传体验和大文件处理思路。
- `filebrowser`：参考工作目录浏览和文件管理 UI 组织方式。
- `Pingvin Share`：参考分享配置语义；仅作设计参考，不作为长期依赖。
- `Directus`、`Strapi`：参考内容建模、字段组织和后台视角。

### 6.3 复用结论

- 不直接复用单一开源项目交付全部需求。
- 采用“底座固定为 `PocketBase`，其他项目按能力局部借鉴”的复用策略。

## 7. 明确采用项

- 采用 `PocketBase` 作为底座。
- 采用自研薄业务壳承接本项目特有规则。
- 采用自研网页层承接对外内容访问和列表搜索展示。
- 采用统一内容模型同时覆盖文件与 HTML 富文本。
- 采用 hash URL 作为对外访问标识。

## 8. 明确不采用项

- 不采用 `Appwrite` 作为本轮底座。
- 不采用 `Supabase` 作为本轮底座。
- 不采用 `Directus`、`Strapi`、`NocoDB`、`Wiki.js` 作为主承载系统。
- 不采用 `Gokapi`、`chibisafe`、`Pingvin Share`、`filebrowser` 作为主承载系统。
- 不采用 `Halo`、`Memos` 作为主承载系统。
- 不采用“一个现成开源项目直接覆盖全部需求”的路线。

## 9. 带入 Architecture 的技术约束

- 架构必须围绕 `PocketBase + 业务壳 + 对外访问层` 三层组织。
- 文件与 HTML 富文本必须进入统一内容模型，而不是分别走独立系统。
- 业务壳必须掌握 API Key 映射、内容归属、分享访问和 hash 路由规则。
- 对外访问层必须与后台管理层职责分离。
- 运行时复杂度必须适配 MiniPC 单机部署，不引入重型组件栈。
- MVP 架构应优先打通基础上传、保存、列表、搜索、详情和分享访问主链路。

## 10. 待确认事项

- `PocketBase` 以独立服务运行，还是嵌入到自定义 Go 应用中。
- 业务壳和网页层的具体技术栈。
- HTML 内容的安全清洗与渲染策略。
- 搜索是否只做基础数据库查询，还是预留全文检索扩展。
- 工作目录与底座文件存储之间的落盘关系。
