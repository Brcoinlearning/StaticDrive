# 调研成果讲解

## 开场

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 1 章到第 2 章。

这次调研我们不是在找一个功能最多的开源项目，而是在回答三个问题：第一，这类业务应该优先选哪条技术路线；第二，在这条路线里哪个项目最适合作为底座层；第三，哪些能力适合参考其他开源项目，再补一层很薄的业务壳。

从需求上看，这个系统的本质不是普通 CMS，也不是普通网盘，而是一个运行在 `1Panel + MiniPC` 上、由本地智能体通过 Open API 写入文件和 HTML，再对外提供 HTTP 访问的本地内容中转与发布服务。

## 需求重点

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 2 章。

真正影响选型的点主要有几个：

- 必须支持 Open API 写入文件和 HTML。
- 不只是上传后给一个链接，还要支持列表、搜索、详情。
- 后续还要承接用户归属、分享访问、Hash URL 和生命周期管理。
- 部署环境受限，必须考虑 MiniPC 下的轻量化。

所以我们的判断标准不是“功能多不多”，而是“谁更适合做这个本地子系统的底座”。

## 调研收敛

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 3 章到第 4 章。

我们先没有直接比较单个项目，而是先按大类分成四类：轻量后端、Headless CMS、文件分享类产品、建站/笔记类产品。

然后先做大类判断。结论是，这个项目的主路线应该优先选择“轻量后端底座”，而不是直接选 CMS、建站工具或者文件分享产品。

原因是：建站和笔记类前台强，但系统入口和内容模型不对；文件分享类很适合做外链和上传，但统一内容模型、搜索和后续扩展不够自然；Headless CMS 能力很强，但对当前 `MiniPC + 1Panel` 的边缘场景偏重。

## 底座结论

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 5 章。

在轻量后端这条赛道里，我们重新比较了 `PocketBase`、`Appwrite`、`Supabase`。

最终判断是 `PocketBase` 最适合作为底座层。不是因为它功能最全，而是因为它更轻、单机部署更友好，而且自带文件、用户、API、后台这些基础能力，同时还适合后续补一层业务壳。

相对来说，`Appwrite` 和 `Supabase` 都更像完整平台，能力更大，但对当前这个本地子系统来说边界偏重。

## 最终方案

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 6 章到第 8 章。

最后我们的成果不是“选一个项目直接交付”，而是确定了一个分层方案：

- 底座层选 `PocketBase`。
- 参考设计层吸收 `Gokapi`、`chibisafe`、`filebrowser` 这些项目在分享治理、上传体验、文件管理上的成熟设计。
- 然后只自研一层很薄的业务壳，去承接我们自己的规则，比如用户归属、Hash URL、分享校验、搜索接口这些。

所以文件分享类项目不是没用，而是我们把它们放到了参考设计层，而不是让它们直接承担底座层。

## 后续待确认

讲到这里，看 [技术选型报告](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/《“书童四九”本地文件管理服务技术选型调研报告》) 第 9 章到第 10 章。

调研到这一步，技术路线已经比较清楚了。后面还需要业务侧继续确认几个边界：

- 用户模型第一阶段做到什么程度。
- 分享链接在 MVP 阶段要做到多强，比如是否立刻支持撤销、过期。
- 生命周期管理是先预留，还是第一阶段就落自动清理。
- 大文件支持是否第一阶段就要做优化。

这些边界一旦定下来，就可以进入 MVP 的架构设计和实现拆解。
