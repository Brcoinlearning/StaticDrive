# 《“书童四九”本地文件管理服务技术选型调研报告》

## 一、调研目的与评估边界

本文基于 [业务规则备忘录](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/20-planning/业务规则备忘录.md) 对候选开源项目进行技术选型评估。评估目标不是简单判断“项目是否优秀”，而是回答以下问题：

1. 候选项目是否能在 1Panel 管理的 MiniPC 环境中稳定部署。
2. 是否能为“书童四九”提供可编程的文件与富文本写入能力。
3. 是否能支撑“对外 HTTP 访问 + URL Hash 隔离 + 用户归属 + 分享访问”的业务模型。
4. 若不能原生满足，需要补哪些自研模块，补齐成本是否可接受。

需要特别说明的是，本报告将“业务需求满足度”和“原生能力”分开评估。某方案即使可以通过二次开发落地，也不等于它原生满足需求。

## 二、业务需求基准

依据 [业务规则备忘录](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/20-planning/业务规则备忘录.md) 与 [用户原始需求.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/用户原始需求.md)，本项目的关键约束如下。

### 2.1 环境与架构约束

- 部署环境为 1Panel 管理的 MiniPC。
- 需要支持容器化部署，便于接入 1Panel 的反向代理、证书和备份能力。
- 系统定位是“文件管理服务 + 静态网页服务”，不是通用内容平台。

### 2.2 Must 级业务能力

- `FR-001` 文件上传与本地工作目录存储。
- `FR-002` 富文本 HTML 保存与渲染。
- `FR-003` 内容列表展示，且用户只能看到自己的内容。
- `FR-004` 文件名/标题搜索。
- `FR-005` 内容详情查看与下载；对外 URL 使用 Hash 是明确需求，但在原始需求中优先级低于基础可访问能力。
- `FR-006` 内容 CRUD。
- `FR-007` 分享链接生成与撤销。
- `FR-008` 面向智能体的 Open API，使用 API Key 识别身份。

### 2.3 核心业务规则

- `BR-001` 所有内容必须归属于某个用户，API Key 是归属识别入口。
- `BR-002` 用户只能管理自己的内容，别人只能通过分享链接访问。
- `BR-003` 对外 URL 使用 Hash 且不暴露原始文件名；但从原始需求看，该项可视为 MVP 之后优先补齐的增强约束。
- `BR-004` 分享链接至少要支持撤销，后续可扩展一次性、过期时间、密码访问。
- `BR-005` 搜索必须覆盖文件名，标题搜索是加分项。

此外，原始需求还额外说明了两点范围信息：

- 大文件（如音视频）支持是低优先级，不应反过来主导本轮选型。
- 文件生命周期管理是明确需求，但优先级低于基础的上传、展示、搜索与 Open API。

结论上，这不是一个“单纯附件上传”场景，而是一个带有 API 写入、内容发布、静态展示和后续可扩展治理能力的轻量内容发布服务。若严格采用业务备忘录口径，还需要兼容用户归属与分享访问模型。

## 三、候选项目原始材料摘要

本次评估的依据，来自仓库内整理的项目介绍材料：

- [memos-readme.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/memos-readme.md)
- [pocketbase-readme.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/pocketbase-readme.md)
- [directus-introduction.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/directus-introduction.md)
- [halo-readme.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/halo-readme.md)

根据这些材料，可以确认以下事实。

### 3.1 Memos

- 定位是自托管笔记服务。
- 技术栈是 Go 后端 + React 前端。
- 提供 REST 与 gRPC API。
- 核心数据形态强调 Markdown 原生与纯文本存储。

### 3.2 PocketBase

- 定位是轻量级 Go 后端 / BaaS。
- 内置 SQLite、文件管理、用户管理、管理后台和 REST-ish API。
- 支持作为独立应用运行，也支持作为 Go 库嵌入自定义业务逻辑。

### 3.3 Directus

- 定位是运行在 SQL 数据库之上的实时 API 和应用仪表板。
- 核心能力是 REST / GraphQL API、可扩展平台、现代后台面板。
- 技术栈是 Node.js + TypeScript + Vue。

### 3.4 Halo

- 定位是功能完整的开源建站工具。
- 技术栈是 Java + Spring Boot。
- 官方明确推荐结合 1Panel 部署。
- 优势在于站点能力、插件与主题生态，而不是轻量 BaaS。

## 四、评估维度

为了避免“只看某一个亮点”造成误判，本报告采用以下五个维度进行评估：

1. `部署适配度`：是否适合在 MiniPC + Docker + 1Panel 下运行。
2. `数据接入能力`：是否便于承接文件上传、HTML 存储、Open API 接入。
3. `业务规则承载能力`：是否适合实现用户归属、列表隔离、分享访问、Hash URL。
4. `展示层补齐成本`：是否需要额外开发列表页、详情页、搜索页和下载链路。
5. `总体改造成本`：为了满足业务需求，需要补多少自研逻辑。

## 五、候选方案逐项分析

### 5.1 Memos

#### 优势

- Go 后端，部署轻量。
- 自带前端界面，启动成本低。
- 原生提供 REST + gRPC API，集成门槛不高。

#### 与业务需求的主要偏差

- 原始材料强调的是 Markdown 原生与纯文本存储，这与 `FR-002` 要求的“接收 HTML 并按 HTML 形式渲染”存在天然错位。
- 它的产品模型更接近笔记流，而不是“文件 + 富文本 + 分享链接 + 按用户隔离的静态发布服务”。
- 即使 API 可用，也无法从项目定位上证明它适合承载 `BR-003` 的 Hash URL、`FR-007` 的分享撤销以及 `FR-006` 的内容管理闭环。

#### 结论

Memos 只能证明“可以存文字并提供 API”，不能证明“适合作为本地文件托管与发布底座”。它与本项目需求的核心数据模型不匹配，建议淘汰。

### 5.2 PocketBase

#### 优势

- Go 单体后端 + SQLite，最符合 MiniPC 轻量化目标。
- 原生提供文件管理、用户管理、管理后台和 REST-ish API。
- 支持作为 Go 库嵌入，这意味着可以在同一技术栈下补自定义路由和业务逻辑。

#### 已满足或较易满足的需求

- `FR-001` 文件上传与元数据保存：适合。
- `FR-002` HTML 文本保存：可通过自定义字段实现。
- `FR-008` Open API：原生 API 能力较强。
- 容器化与单机部署：适合。

#### 需要补齐的关键能力

- `FR-003` 列表展示和 `FR-005` 详情页：PocketBase 只有后台 UI，没有对外展示页。
- `FR-004` 搜索：需要基于集合字段自行设计查询接口和前端交互。
- `FR-006` CRUD：基础数据操作可做，但要结合用户归属和业务规则封装接口。
- `FR-007` 分享链接与撤销：需要自建分享表或分享映射逻辑。
- `BR-003` Hash URL：默认文件访问路径不等于业务要求的对外 Hash 路径，必须增加代理层或自定义路由层。
- `BR-001` / `BR-002`：虽然有用户管理，但“API Key 对应业务用户”“用户只能看自己的内容”仍需要明确建模和鉴权设计。

#### 结论

PocketBase 不是“开箱即用即满足全部业务”的方案，但它是最适合作为底层数据与文件能力底座的方案。若接受“PocketBase 负责数据底座，自研一层轻量业务/展示服务”，则整体最平衡。

### 5.3 Directus

#### 优势

- 原生 API 能力强，适合管理结构化内容。
- 对 SQL 数据模型的表达能力明显强于通用笔记工具。
- 适合作为管理后台和内容建模平台。

#### 与需求的契合点

- 适合承接 `FR-001`、`FR-002`、`FR-008` 这类“数据接入型”需求。
- 在权限、字段建模、后台管理体验上，比 PocketBase 更成熟。

#### 主要问题

- 原始材料显示其核心栈为 Node.js 平台，并常与更完整的外围组件一起部署。对 MiniPC 来说，静态资源消耗与运维复杂度高于 Go 方案。
- 同样缺少面向外部用户的成品静态展示页面，`FR-003`、`FR-005`、`FR-007` 仍需补前台。
- 即使文件访问路径可做成非原文件名形式，也不能直接推导出它天然满足“分享撤销、业务 Hash、外部访问治理”这整套规则。相关能力仍需结合业务层实现。

#### 结论

Directus 更像“能力很强的内容平台”，而不是“边缘场景下的最小文件发布服务”。如果后续业务复杂度显著上升、且 MiniPC 资源不是主要约束，可以作为备选；若当前强调轻量和简单，优先级低于 PocketBase。

### 5.4 Halo

#### 优势

- 官方推荐结合 1Panel 部署，运维接入最顺滑。
- 自带完整站点能力，前台展示、主题、内容呈现成熟。
- 对“尽快搭一个可访问站点”这件事很友好。

#### 与需求的主要偏差

- Halo 的产品定位是建站工具，而不是面向智能体写入的轻量文件服务底座。
- Java + Spring Boot 对 MiniPC 资源更不友好。
- 它虽然适合做内容网站，但“用户只能看自己的内容列表”“外部只能通过分享链接访问”“对外 URL 必须是业务 Hash 且不暴露原文件名”这类约束，并不能从其原始材料中证明能低成本实现。
- 若为了满足 `FR-008`、`BR-001`、`BR-003` 去对接或改造其内部模型，成本和耦合度都偏高。

#### 结论

Halo 适合“先搭站”，不适合“先做一个轻量 API 文件中转与发布服务”。如果业务目标是 CMS 站点，它很合适；如果目标是书童四九的子系统底座，则不推荐作为主方案。

## 七、推荐结论

### 7.1 主推荐方案

**PocketBase 作为底座 + 自研轻量业务/展示层**，仍是当前最稳妥的主推荐方案。

推荐理由：

- 它最符合 MiniPC 轻量化要求。
- 它原生具备文件管理、用户管理、REST-ish API 和 SQLite，足以承接智能体写入链路。
- 它支持作为 Go 库嵌入，便于后续把业务 Hash 路由、分享链接校验、列表与详情接口统一放进一个轻量服务内。
- 相比 Directus 和 Halo，它更接近“文件服务底座”而不是“完整内容平台”。

如果只追求“尽快上线一个可访问站点”，且暂不严格执行 Hash URL、用户隔离与分享治理，Halo 可以视为一个更快的妥协型方案；但这不是本报告的主推荐路径。

### 7.2 不建议直接下结论的点

以下表述不应再使用：

- “PocketBase 方案完全符合全部硬性与弹性要求。”
- “Directus 原生就完整满足 Hash 安全需求。”
- “Halo 只要部署好就几乎零改造满足需求。”

更准确的说法应该是：

- PocketBase 是**底座最合适**，但仍需补业务层。
- Directus 是**能力更强但更重**的备选。
- Halo 是**建站能力强但系统定位偏离**的方案。

## 八、建议落地架构

若采用主推荐方案，建议拆成两层，而不是把所有责任压在某个开源项目上。

### 8.1 数据底座层

使用 PocketBase 承担：

- 文件落盘
- 元数据存储
- 基础后台管理
- 基础 API 能力

### 8.2 业务与展示层

自研一个轻量服务，承担：

- API Key 到业务用户的映射
- 内容归属校验
- 用户列表、搜索、详情、删除接口
- 分享链接生成、撤销、校验
- `/files/{hash}.{ext}` 或 `/share/{hash}` 的对外路由
- HTML 渲染模板与 XSS 清洗
- 文件流式下载转发

这样可以把“轻量文件底座”和“业务规则执行”分开处理，符合当前项目的实际边界。

## 九、实施风险提示

### 9.1 不要把“有后台”误当成“满足业务前台”

候选项目里的后台 UI 主要面向管理者，不等于业务要求里的用户列表页、搜索页、详情页和分享访问页。

### 9.2 不要把“非原文件名 URL”误当成“完整分享安全模型”

业务要求的不只是 URL 漂亮或隐藏文件名，而是要把所有权、分享、撤销和外部访问链路一起管起来。

### 9.3 富文本 HTML 渲染必须显式处理安全问题

若内容来自大模型输出，展示前必须经过 HTML 安全清洗，否则存在 XSS 风险。

### 9.4 大文件下载必须走流式转发

MiniPC 资源有限，代理层不应将大文件整体读入内存再返回。

---

## 十、最终结论

基于当前仓库中的业务需求和候选项目原始材料，可以得出以下结论：

1. 没有任何一个候选开源项目能够原生完整满足全部业务需求。
2. Memos 与项目数据模型不匹配，应淘汰。
3. Halo 的强项是建站，不是轻量文件服务底座，不建议作为主选型。
4. Directus 能力强，但对当前边缘场景偏重。
5. PocketBase 最适合作为底层数据与文件能力底座，但必须承认仍需补一层轻量业务/展示服务，才能真正满足业务规则备忘录。

因此，当前最合理的选型结论不是“选一个开源项目直接交付”，而是：

**选择 PocketBase 作为底座，并明确将用户归属、分享治理、Hash 路由、静态展示和搜索能力纳入自研范围。**
