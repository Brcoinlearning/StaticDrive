# 核心业务链路图白话版

本文档是 [core-business-flow.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/core-business-flow.md) 的白话讲解版。

目标不是讲源码细节，而是把每条链路用“技术小白也能看懂”的方式讲清楚。

说明规则：

1. 章节顺序完全对照原始链路图。
2. 新技术词只在第一次出现时解释。
3. 后面再出现同一个词，只继续讲业务含义，不重复讲概念课。

## 1. 总览图白话解释

先把整个系统想成一个“文件与内容管理柜台”。

这里面有四个主要角色。

### 1.1 调用方

就是来使用系统的人或程序。

它可能做这些事：

- 上传 PDF 或别的文件
- 上传一段 HTML 富文本
- 查看自己的内容
- 把内容分享给别人

### 1.2 业务壳服务 Node

这是整个系统真正的“前台柜员”。

它负责：

- 接收请求
- 判断你有没有权限
- 决定这次请求该怎么处理
- 去找数据库或本地文件
- 最后把结果返回给你

你可以把它理解成：大脑和总调度中心都在这里。

### 1.3 PocketBase

这是系统里的“登记簿 + 档案卡系统”。

它不一定保存真实文件本体，但会保存很多管理信息，比如：

- 这个用户是谁
- 这条内容属于谁
- 是否已经公开
- 分享链接是什么
- HTML 正文是什么

简单说，PocketBase 更像“管理台账”。

### 1.4 workspace/content-files/

这是本地文件仓库。

像 PDF、TXT 这种真正的文件内容，会放在这里。

所以你可以把系统理解成：

- 业务壳负责办事
- PocketBase 负责记账
- `workspace/content-files/` 负责真正存文件

## 2. 文件写入链路白话解释

这一条链路对应的是：上传一个 PDF、TXT、Markdown 之类的真实文件。

### 2.1 你先把文件交给系统

你调用的是 `POST /api/write/file`。

你要交给系统几样东西：

- `API Key`
- `title`
- `filename`
- `mimeType`
- `contentBase64`

这里先解释两个第一次出现的技术词。

### 2.2 什么是 API Key

你可以把 `API Key` 理解成“这次办事时出示的身份凭证”。

它的作用是告诉系统：

- 你是谁
- 你有没有资格写入内容
- 这条内容最后应该记到谁名下

如果没有这个凭证，系统就不知道该把文件归给谁。

### 2.3 什么是 base64

文件原本是二进制内容，直接塞进 JSON 这种文本结构里不方便。

所以这里用了 `base64`。你可以把它理解成：

“先把文件临时翻译成一长串普通文本字符，再交给接口传输。”

这样做的好处是：

- 接口更统一
- 请求体更稳定
- 不需要另外上一套复杂的文件上传协议

你不用把它理解成加密，它只是“为了方便传输而换一种表示方式”。

### 2.4 系统先确认你是谁

业务壳拿到请求后，第一件事不是存文件，而是先看 `API Key` 是否有效。

为什么要先做这个？

因为系统必须先知道“这份文件属于哪个 owner”。

否则就会出现两个问题：

- 谁都能乱传文件
- 传进来的文件不知道该记到谁名下

### 2.5 系统检查文件内容能不能正常解开

它会把 `contentBase64` 还原回真正的文件内容。

这一步是在确认：

- 你传来的内容是不是合法的
- 文件有没有损坏
- 这次上传是不是空壳

你可以把它理解成“先拆包验货”。

### 2.6 系统整理文件名

上传上来的文件名不一定适合直接拿去存磁盘。

比如有些文件名可能：

- 带奇怪字符
- 带路径符号
- 过于混乱

所以系统会先把文件名清洗成更安全、更适合保存的样子。

这一步就像仓库管理员收到一个标签乱七八糟的箱子，先把标签改规范再入库。

### 2.7 系统生成 content_hash

这里再解释一个第一次出现的技术词：`hash`。

在这个项目里，`content_hash` 你可以先不用把它想得太复杂。它主要就是“给这条内容分配的唯一编号”。

这个编号有三个主要作用：

1. 以后对外访问时，可以靠它定位内容。
2. 保存文件时，可以拿它参与拼路径，减少重名冲突。
3. 这个编号很难猜，比简单的 `1、2、3` 更安全一点。

所以这里的 `hash`，你可以理解成“唯一取件码”。

### 2.8 系统计算 storage_path

`storage_path` 就是“这个文件最后放在仓库里的具体路径”。

为什么不把所有文件都丢到一个目录里？

因为那样文件一多会非常乱，也更难管理。

所以系统会根据 `content_hash` 把文件分散到不同子目录下，再生成一个完整路径。

你可以把它理解成：

- `content_hash` 是档案编号
- `storage_path` 是档案柜里的具体格子位置

### 2.9 先把真实文件写到本地仓库

这一步会把 PDF、TXT 这类真实文件，写进 `workspace/content-files/`。

为什么文件本体不直接全放数据库里？

因为数据库更适合保存管理信息，而文件系统更适合保存真正的文件内容。

简单理解就是：

- 数据库像“档案卡系统”
- 磁盘像“真正放档案盒的仓库”

### 2.10 再把元数据写到 PocketBase

文件已经进仓库后，系统还要在 PocketBase 里登记一条记录。

这条记录就是这份文件的“档案卡”。

它会记录：

- 这是哪个 owner 的内容
- 类型是 `file`
- 标题是什么
- 原文件名是什么
- `content_hash` 是什么
- `storage_path` 是什么
- 文件类型和大小是什么
- 现在是不是公开状态

为什么还要多做这一步？

因为后续 owner 页面、搜索、详情、分享、删除，不是靠直接扫磁盘目录完成的，而是先靠数据库里的这张档案卡管理。

### 2.11 为什么要有失败补偿

如果文件已经写到磁盘了，但写 PocketBase 失败了，会怎样？

那就会出现一种很麻烦的脏状态：

- 仓库里有文件
- 数据库里没登记
- 页面上看不到它
- 但它实际还占着空间

所以系统会做一个补救动作：

- 如果磁盘写成功了
- 但数据库写失败了
- 就把刚写进去的文件删掉

这样两边才不会长期不一致。

### 2.12 这条链路最后的结果

如果一切顺利，系统会返回：

- `contentId`
- `contentHash`
- `accessUrl`

这表示：

- 真实文件已经放到本地仓库
- 数据库档案卡也已经建好
- 后续系统已经知道该怎么查、怎么显示、怎么分享它

## 3. HTML 富文本写入链路白话解释

这一条链路对应的是：上传一段 HTML 内容，而不是上传一个真实文件。

你可以把它理解成“存一篇网页内容”，而不是“存一个 PDF 附件”。

### 3.1 请求进入系统

你调用的是 `POST /api/write/html`。

这次带的是：

- `API Key`
- `title`
- `htmlContent`

这里的 `API Key` 和前一条链路作用完全一样，还是先确认“你是谁”。

### 3.2 系统先确认 owner 身份

这一步和文件上传没有本质区别。

业务壳先确认：

- 这个 key 合不合法
- 它对应哪个 owner

### 3.3 系统检查标题和 HTML 正文

系统会检查：

- `title` 不能为空
- `htmlContent` 必须真的是一段字符串

为什么要查这个？

因为 HTML 这类内容没有单独文件名、没有磁盘文件本体，所以标题和正文就是它最核心的数据。

### 3.4 系统也会生成 content_hash

虽然这里不是文件，但系统仍然给它生成一个唯一编号。

原因很简单：

- 它也要有自己的内容地址
- 它也要能被查询、分享、公开访问
- 系统希望“文件”和“HTML”都放进同一个统一内容模型里

你可以把它理解成：

不管你交来的是 PDF 还是一篇 HTML，系统都先给你发一个统一编号。

### 3.5 和文件链路最大的差别

这里最重要的区别只有一个：

- 文件类型：真实内容放在 `workspace/content-files/`
- HTML 类型：真实内容直接放在 `PocketBase.contents.html_content`

所以 HTML 不会在 `workspace/content-files/` 下再生成一个 `.html` 文件。

这就是为什么你前面看到：

- PDF 能在 `workspace` 里找到
- HTML 找不到对应本地文件

因为它本来就不是这么存的。

### 3.6 为什么 HTML 不落成物理文件

因为这类内容本质上更像“一段正文”，而不是一个必须单独管理的附件文件。

如果把 HTML 直接放进数据库字段里，系统做这些事会更直接：

- owner 查看详情
- public 页面渲染
- 更新 HTML 正文

如果反过来把每段 HTML 都存成独立文件，管理上反而会更绕。

### 3.7 这条链路最后的结果

最后系统同样会返回：

- `contentId`
- `contentHash`
- `accessUrl`

也就是说，从外部看，文件和 HTML 都被当成统一的“内容”。

只是它们的本体保存位置不同：

- 文件本体在磁盘
- HTML 本体在数据库字段里

## 4. Owner 查询与管理链路白话解释

这一条链路讲的是：owner 登录以后，为什么只能看到自己的内容，以及列表页、搜索页、详情页是怎么来的。

### 4.1 owner 先登录

浏览器先打开 owner 登录页，然后提交自己的 `API Key`。

这里和前面的 API 写入链路不一样的地方在于：

- API 场景通常是每次请求都自己带 `API Key`
- 浏览器场景更适合“登录一次，后面自动记住身份”

### 4.2 什么是 session

这里第一次出现 `session` 这个词。

你可以把它理解成“登录成功后，服务端临时给浏览器发的一张通行证”。

有了这张通行证，后面浏览器再访问 owner 页面时，不需要你每次手工再输一遍 `API Key`。

它的作用和商场会员手环有点像：

- 第一次登记身份
- 后面只要手环还有效，就默认知道你是谁

### 4.3 登录成功后，浏览器后续带着 session 来访问

接下来 owner 打开的页面，常见有：

- `/web/list`
- `/web/search`
- `/web/detail/:contentId`

业务壳会先看 session，确认当前浏览器对应哪个 owner。

### 4.4 为什么 owner 只能看到自己的内容

很多人会以为这是页面层做了“隐藏别人的数据”。

其实不是。

真正发生的是：业务层查询数据库时，就已经只按当前 owner 去拿数据了。

也就是说：

- 不是先把所有人的内容都拿出来，再在页面上遮住别人的
- 而是一开始就只查“属于当前 owner 的内容”

这会更安全，也更符合业务规则。

### 4.5 列表、搜索、详情分别在做什么

列表页：

- 按当前 owner 查询内容
- 返回一批内容摘要

搜索页：

- 仍然只在当前 owner 的内容范围里搜索
- 不是全站搜索

详情页：

- 先按 `contentId` 找到那条内容
- 再确认它是不是当前 owner 的
- 只有两者都满足才返回

### 4.6 为什么还要做 owner 权限检查

因为只靠“页面上没有入口”不够安全。

即使某人手动拼了一个别人的 `contentId`，业务层也会再检查一次归属。

如果不是自己的，就不给看。

所以真正的边界在业务层，不在页面按钮上。

## 5. 分享创建与公开访问链路白话解释

这一条链路分两半：

1. owner 怎么创建分享
2. 外部访客怎么通过公开地址访问

### 5.1 创建分享

当 owner 选择“分享”某条内容时，系统做的不是“把文件复制一份发出去”，而是给这条内容创建一个公开访问入口。

### 5.1.1 什么是 share_link

这里第一次出现 `share_link`。

你可以把它理解成：

- 一条分享记录
- 里面有一个分享用的取件码 `share_hash`
- 这个取件码可以生成一个分享地址

所以分享这件事，本质上是“新建一条可公开访问的门票记录”。

### 5.1.2 系统为什么先检查 owner 权限

因为只有内容拥有者，才有资格决定是否公开这条内容。

所以系统会先确认：

- 这条内容是不是你的
- 你有没有权利给它开分享

### 5.1.3 如果之前已经分享过怎么办

系统会先查这条内容有没有还有效的分享记录。

如果已经有：

- 不会重复造一条新的
- 会直接复用原来的 `share_hash`

为什么这样做？

因为这样可以避免一条内容被反复点“分享”后生成很多重复入口。

### 5.1.4 contents.is_shared 是什么

这里第一次出现 `is_shared`。

你可以把它理解成“这条内容现在是不是处于公开状态”的开关。

当分享建立完成时，这个开关会被改成 `true`。

### 5.2 公开访问

外部访客访问公开内容时，有两种方式。

第一种：用 `contentHash` 访问。

第二种：用 `shareHash` 访问。

你可以把它理解成：

- `contentHash` 像内容自己的编号入口
- `shareHash` 像分享门票入口

### 5.2.1 按 contentHash 访问时系统怎么想

系统会先查：

- 有没有这条内容
- 这条内容现在是不是公开的，也就是 `is_shared` 是否为 `true`

只有公开状态成立，才允许外部访问。

### 5.2.2 按 shareHash 访问时系统怎么想

系统会先查分享记录，再看这张“门票”还有效不有效。

这里第一次出现 `is_revoked`。

你可以把它理解成“这张分享门票有没有被作废”的状态。

- `false` 表示还有效
- `true` 表示已经作废

如果门票已经作废，就不允许继续访问。

### 5.2.3 为什么公开访问最后还要分 file 和 rich_text

因为这两类内容的本体保存方式不同。

如果是 `rich_text`：

- 直接从数据库里拿 HTML 正文
- 然后返回给页面渲染

如果是 `file`：

- 需要去本地文件系统把真实文件读出来
- 再把真实字节流返回给访客下载

也就是说，外面看起来都是“访问内容”，但系统内部处理方式不同。

## 6. 删除与撤销分享链路白话解释

这一条也分两半：

1. 撤销分享
2. 删除内容

### 6.1 撤销分享

撤销分享不是删除内容本身，而是把“对外公开通道”关掉。

### 6.1.1 系统做了什么

owner 发起撤销分享后，系统会：

1. 先确认你是不是这条内容的 owner
2. 找到当前有效的分享记录
3. 把 `share_links.is_revoked` 改成 `true`
4. 把 `contents.is_shared` 改成 `false`

你可以把它理解成两件事一起发生：

- 门票作废
- 内容公开开关关闭

### 6.1.2 为什么访问结果会变成 410 Gone

这里第一次出现 `410 Gone` 这个 HTTP 状态。

你可以把它理解成：

“这个地址以前是存在的，但现在已经被明确作废了。”

它和“压根不存在”不完全一样。

系统用这个状态是在表达：

- 不是没找到
- 而是这个分享入口以前有，现在被撤销了

### 6.2 删除内容

删除内容比撤销分享更彻底。

它不是只关掉公开入口，而是要把这条内容整个移除。

### 6.2.1 系统删除内容时的顺序

系统会这样做：

1. 确认你是不是 owner
2. 找出这条内容所有相关的分享记录
3. 先把还活着的分享入口都作废
4. 如果这是文件类型，删除磁盘里的真实文件
5. 删除数据库里的内容记录

### 6.2.2 为什么先撤销分享，再删内容

因为分享入口本质上还是指向这条内容。

如果不先处理分享记录，就可能出现：

- 内容删了
- 旧分享入口却还残留着

先把分享作废，能让系统状态更干净。

### 6.2.3 删除文件类型和删除 HTML 类型有什么区别

文件类型：

- 要删数据库记录
- 还要删磁盘里的真实文件

HTML 类型：

- 只需要删数据库里的内容记录
- 因为它本来就没有单独的物理文件

### 6.2.4 删除成功后会发生什么

删除成功后，这条内容会同时从几个地方消失：

- owner 列表里看不到了
- 公开入口失效了
- 如果它是文件，本地仓库里也没了

## 7. 压缩版业务时序总图白话解释

如果把全系统浓缩成最短的一段话，可以这样理解：

### 7.1 写入阶段

用户先证明自己是谁，然后把内容交给业务壳。

业务壳再决定：

- 如果是真实文件，就写进本地仓库，再在数据库建档
- 如果是 HTML，就直接写进数据库建档

最后系统给这条内容一个统一身份：`contentId + contentHash`。

### 7.2 管理阶段

owner 登录后，系统只会给他看属于自己的内容。

他可以：

- 看列表
- 搜索
- 看详情
- 分享
- 撤销分享
- 删除

### 7.3 分享阶段

当 owner 选择分享时，系统不是复制内容，而是新增一个公开入口。

这个入口可以被外部访客使用。

### 7.4 公开访问阶段

外部访客通过公开入口访问时，系统会先判断：

- 这条内容现在是否公开
- 这张分享门票是否还有效

然后再根据内容类型决定：

- rich text 直接返回渲染内容
- file 返回真实下载文件

## 8. 建议的阅读顺序白话解释

如果你以后还想继续往源码里深入，建议按原文档给的顺序看。

为什么这样排？

因为这是从“前台接待”一路看到“后台办事”的顺序。

### 8.1 先看 app.js

因为它是总入口。先看它，最容易明白：

- 哪些 URL 会进来
- 每个请求被分发到哪里

### 8.2 再看鉴权相关

也就是：

- `api-key-auth.js`
- `session-auth.js`

因为只有先弄明白“系统怎么知道你是谁”，后面的 owner 边界才容易看懂。

### 8.3 再看 content/service.js

这是核心业务规则最多的地方。

写入、分享、删除、公开访问，基本都在这里落地。

### 8.4 再看 PocketBase client

因为到了这一步，你再看数据库读写层，就容易明白“业务壳到底让数据库做了什么”。

### 8.5 最后看 page-renderer

因为页面渲染更像“把前面已经算好的结果展示出来”。

如果先看它，容易只看到页面长什么样，却还不理解业务为什么这么走。
