# 《“书童四九”本地文件管理服务技术选型调研报告》

## 一、报告目标与决策边界

本报告的目标，不是寻找一个“功能最多”的开源项目，也不是为现有结论寻找论据，而是基于项目的真实业务约束，回答三个更具体的问题：

1. 本项目的主路线，应该优先选择哪一类开源产品。
2. 在该类产品中，哪个项目最适合作为底座层。
3. 哪些非底座能力应参考其他开源项目的成熟设计，再由我们补上一层很薄的业务壳。

因此，本报告会把“路线选择”“底座选择”“局部能力参考”拆开讨论，而不把所有候选项目放在一个表里平铺对比。

同时，本报告明确遵循以下方法约束：

- 不继承既有调研报告中的推荐结论。
- 所有判断优先依据 [用户原始需求.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/用户原始需求.md) 与 [业务规则备忘录.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/20-planning/业务规则备忘录.md)。
- `PocketBase` 是否胜出，必须经过与同赛道候选的重新比较后才能得出，不作为预设前提。

### 2.2 决策原则

#### 原则一：系统首先是“本地内容中转与发布服务”，不是通用站点系统

本项目运行在 `1Panel + MiniPC` 的单机环境中，服务对象是本地智能体“书童四九”和外部访问用户。核心链路是：

`智能体生成内容 -> 文件管理服务接收和落盘 -> 静态网页服务对外提供只读访问`

这决定了系统本质更接近一个“本地内容与文件发布底座”，而不是博客、知识库或内容运营平台。

#### 原则二：Open API、文件落盘、HTML 内容存储是第一优先级

无论底座层怎么选，至少必须稳定承担以下能力：

- 通过 Open API 接收文件上传
- 通过 Open API 接收 HTML 富文本内容
- 将文件保存到本地工作目录
- 保存文件/内容元数据
- 为后续列表、搜索、详情提供统一的数据基础

换言之，底座首先要解决的是“写入”和“存储”问题，而不是主题系统、内容美观展示或复杂运营能力。

#### 原则三：列表、搜索、详情是核心能力，不是可选附加项

从原始需求到业务备忘录，`内容列表`、`内容搜索`、`内容详情查看` 都是主流程的一部分。这意味着候选系统不能只会“上传后给个链接”，还要支持统一内容索引和面向内容消费的查询能力。

#### 原则四：用户归属、分享访问、Hash URL 更像业务层规则，而非必须由底座原生包办

`API Key 识别用户`、`内容归属`、`只能看自己的列表`、`分享访问`、`Hash URL` 都非常重要，但这些更像业务语义，而不是所有底座都会原生提供的能力。

因此，评估时不能简单以“底座是否原生具备全部业务规则”来一票否决，而要看：

- 该底座是否适合承载这些规则
- 为补齐这些规则需要增加多少自研工作
- 补齐方式是否清晰且可控

#### 原则五：应控制复杂度，而不是机械追求“越轻越好”

原始需求已经说明：

- 大文件（如音视频）支持优先级较低
- 生命周期管理明确存在，但低于基础上传、发布、搜索和访问链路
- Hash URL 也是明确需求，但优先级低于“先形成安全可用的访问链路”

同时，`MiniPC` 并不等于“性能极弱”，因此本项目并不是单纯追求“最小化”本身，而是追求：

- 复杂度可控
- 运维成本可控
- 能力边界与当前问题结构匹配

因此，这些能力应该进入架构设计，但不应反过来把底座选型带偏到“只选最擅长大文件外链的产品”或“只选看起来最小的产品”。

## 四、为什么主路线应选择“轻量后端底座”

### 4.1 为什么不是建站 / 笔记类产品作为主路线

以 `Halo`、`Memos` 为代表的产品，优势在于现成前台、主题或内容呈现能力，但这类产品的系统中心通常是“站点”或“笔记产品”，而不是“可被智能体驱动写入的本地内容底座”。

这类路线的问题，不在于它们不能展示内容，而在于系统重心放错了位置。

本项目最先要解决的是：

- 智能体如何通过 API 持续写入文件和 HTML
- 数据如何稳定落到本地工作目录
- 文件与富文本如何被统一建模、统一检索、统一访问
- 后续如何加入用户归属、分享访问、Hash URL 等业务规则

而建站 / 笔记类产品通常优先解决的是：

- 页面如何展示
- 内容如何组织成站点结构
- 后台如何面向人工运营
- 主题、插件、站点体验如何完善

这就带来三个实质性问题。

第一，`系统入口错位`。本项目的主要写入方不是人工编辑器，而是本地智能体的 Open API 调用。若底层采用建站 / 笔记模型，往往需要把“文章”“页面”“笔记”等原生概念强行改造成“智能体写入的内容对象”，这会让数据模型从一开始就带着历史包袱。

第二，`内容模型错位`。本项目需要统一承接“二进制文件 + HTML 富文本 + 元数据 + 用户归属 + 分享访问”这组对象关系。建站 / 笔记类产品虽然也处理内容，但它们默认围绕文章流、页面流或 Markdown 流组织，不天然适合作为文件服务底座。

第三，`演进路径错位`。这类产品前台能力通常很强，但未来一旦需要强化 API、归属规则、分享治理、Hash 路由或生命周期管理，系统改造往往不是“往上加一层壳”，而是“持续绕开原生模型”。这类改造短期能做，长期会越来越别扭。

因此，`D 类` 的合理定位不是主路线，而是：

- 作为“快速做出可浏览站点”的妥协参考
- 作为“为什么前台成品能力不等于底座适配度”的对照组

### 4.2 为什么不是文件分享类产品作为主路线

`Gokapi`、`chibisafe`、`Pingvin Share`、`filebrowser` 与本项目表面上最像，因为它们都直接处理上传、下载、外链和文件访问。也正因为它们很像，所以更需要把“不选它们做底座层”的理由写清楚。

文件分享类产品的典型中心问题是：

- 如何把文件交给别人
- 如何生成分享链接
- 如何控制访问次数、密码、过期时间
- 如何优化上传和下载体验

而本项目的中心问题其实是另一组：

- 如何把“文件”和“HTML 富文本”放进一个统一内容模型
- 如何让本地智能体稳定地通过 API 写入内容
- 如何让列表、搜索、详情围绕统一内容对象展开
- 如何在此基础上再叠加分享访问、Hash URL 和生命周期规则

这两组问题有交集，但并不相同。若直接把文件分享类项目当作底座层，最容易出现四类问题。

第一，`对象模型过窄`。这类产品通常把“被分享的文件”作为核心对象，而不是“可搜索、可渲染、可管理的统一内容对象”。对纯文件场景没问题，但本项目还需要原生承接 HTML 富文本内容。

第二，`查询模型偏弱`。分享系统通常强调“拿到链接即可访问”，而不是“围绕统一内容库做列表、搜索、详情、分页、排序”。本项目需要的是一个内容服务，不只是一个链接分发器。

第三，`API 角色不同`。这些产品可以有 API，但很多 API 是围绕上传和分享动作设计的，不一定适合作为“本地智能体长期写入的基础后端接口层”。

第四，`扩展边界不理想`。如果未来要加入用户归属、列表隔离、CRUD、生命周期和后续用户系统接口，文件分享产品往往会暴露出“分享逻辑很强，但业务模型不够宽”的天花板。

因此，文件分享类产品最合理的定位不是底座层，而是：

- 参考它们的分享治理模型
- 参考它们的上传体验与下载链路
- 参考它们对过期、限制、直链访问的产品拆法

### 4.3 为什么不是 Headless CMS / 内容平台作为主路线

`Directus`、`Strapi`、`NocoDB` 这类产品看起来也很有吸引力，因为它们在内容建模、权限、后台管理和 API 方面都更成熟，理论上确实可以覆盖本项目的大量需求。

但问题在于，它们解决的是“通用内容平台如何成立”，而本项目要解决的是“单机边缘环境下，一个轻量本地内容子系统如何成立”。这两者的目标函数并不相同。

将 Headless CMS / 内容平台作为主路线，主要有三类代价。

第一，`平台能力过剩`。这类系统的价值建立在更通用、更复杂的内容场景之上，因此天然会带入更多平台级能力、更多后台抽象和更完整的内容管理面。对于当前需求，这些能力并非完全无用，但明显超出问题边界。

第二，`运维与资源代价更高`。本项目明确运行在 `1Panel + MiniPC` 的边缘环境中，这意味着部署复杂度、运行时体积、附加组件数量都是真实约束。这里的重点不是“必须最轻”，而是“不能明显过度设计”。对于这类场景，功能更全并不自动等于更合适。

第三，`底座角色不够纯`。本项目需要的是一个容易被嵌入、被控制、被继续加壳的底座。Headless CMS 更像“已经有完整世界观的平台”，而不是“留白较多、方便贴合业务边界的基础内核”。

这并不是说 `B 类` 项目不优秀。恰恰相反，它们在内容建模与后台组织方面非常值得参考。`Wiki.js` 就是一个典型例子：从补充材料看，它具备页面 API、资产上传、搜索和细粒度权限，是一个很强的知识库 / 内容平台候选。但它的核心对象仍然是 `page`，内容中心仍然是 `markdown/wiki content`，权限体系也围绕 `pages` 与 `assets` 展开。这说明它很强，但强在“成熟内容平台”，而不是“留白较多的底座层”。

因此，如果直接把 `B 类` 项目作为主路线，整体架构大概率会偏向“为了一个本地子系统引入一整套平台”。

因此，`B 类` 更适合承担两个角色：

- 作为“重平台边界”的对照参考
- 作为内容建模和后台字段组织方式的借鉴来源

### 4.4 为什么“轻量后端底座”更符合问题结构

如果回到需求本身，本项目首先需要的是：

- 一个可以本地部署、稳定运行的后端底座
- 能接 Open API 写入文件和 HTML
- 能存元数据、能管用户、能做基础查询
- 能为后续的分享、Hash 路由、搜索和列表展示提供清晰的数据基础

这正是轻量后端 / BaaS 类产品最擅长的区域。它们未必直接提供成熟前台，也未必原生覆盖全部业务规则，但它们通常具备一个更适合本项目的特征组合：

- 数据与文件能力优先于站点能力
- API-first 优先于人工运营后台
- 可嵌入、可加壳优先于完整平台闭环
- 复杂度与问题结构匹配优先于平台完备性

对于本项目而言，这种路线有两个直接收益：

- 先把最核心的写入、存储、查询能力稳住
- 将业务规则和对外访问逻辑放到一层更薄、更可控的自研业务壳中实现

### 4.5 本章结论

从问题结构而不是项目知名度出发，本项目的主路线应优先收敛到 `轻量后端 / BaaS 底座` 赛道。

也就是说，下一步的关键比较不再是“所有候选谁最好”，而是：

**在 `PocketBase`、`Appwrite`、`Supabase` 这类轻量后端底座中，谁最适合作为本项目的底层核心。**

### 5.1 比较维度

结合本项目的需求，底座比较采用以下六个维度：

1. `部署复杂度`：是否适合在 MiniPC 上自托管。
2. `资源压力`：是否需要较多附加组件或运行时。
3. `文件与元数据能力`：是否能稳定承接文件、HTML 和元数据。
4. `API 与用户能力`：是否能为智能体写入与身份识别提供基础设施。
5. `本地子系统适配度`：是否适合作为一个被嵌入到更大项目中的本地子模块。
6. `二次开发可控性`：是否容易在其之上再补一层很薄的业务壳。

### 5.2 Appwrite

#### 优势

根据 [Appwrite-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/Appwrite-README.md)，Appwrite 是一个 end-to-end 平台，覆盖：

- 用户认证
- 数据库
- 文件存储与管理
- Cloud Functions
- 消息能力
- 静态与服务端渲染站点托管
- 多语言 SDK

这说明它在功能完整性上很强，完全有能力承接“文件 + 数据 + 用户 + API”的基本需求。

#### 局限

但同一份 README 也明确表明：

- 它被设计为一组 Docker 微服务
- 它强调的是完整平台能力，而不是单一轻量进程
- 它不仅仅是后端底座，还试图覆盖托管、函数、消息等更大范围能力

对于一般 Web 系统，这些是优点；但对于 `1Panel + MiniPC` 的本地内容子系统，这意味着更高的部署与运维复杂度。换句话说，Appwrite 的问题不是能力不够，而是平台边界明显大于本项目当前所需。

#### 判断

如果目标是建设一个更完整的通用开发平台，Appwrite 很有竞争力；但如果目标是为“书童四九”提供一个轻量、本地、自托管的文件与内容底座，Appwrite 有明显“平台过宽”的倾向。

### 5.3 Supabase

#### 优势

根据 [Supabase-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/Supabase-README.md)，Supabase 提供：

- Postgres 数据库
- 认证与授权
- 自动生成的 REST / GraphQL API
- Storage
- Functions
- Dashboard

从功能覆盖看，它也足以作为底座使用，尤其在数据能力和 API 能力方面非常成熟。

#### 局限

但 README 同时也显示，Supabase 实际上是多个开源组件的组合：

- Postgres
- Realtime
- PostgREST
- GoTrue
- Storage
- pg_graphql
- postgres-meta
- Kong

这意味着 Supabase 的优势来自“组合式平台能力”，也意味着其自托管形态天然更像一个完整平台栈，而不是一个小型单体后端。

对于需要强数据库能力和大规模扩展能力的系统，这种架构很合理；但对当前 MiniPC 场景而言，这种分层和组件数会带来更高的认知成本、运维成本和资源压力。

#### 判断

Supabase 是强平台，不是强轻量。它适合“把系统建立在一套完整数据平台上”，但并不天然适合“在单机边缘环境中嵌入一个小型内容服务子模块”。

### 5.4 PocketBase

#### 优势

根据 [PocketBase-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/PocketBase-README.md)，PocketBase 具备以下特征：

- Go 后端
- 内置 SQLite
- 内置文件管理和用户管理
- 管理后台 UI
- REST-ish API
- 可独立运行
- 也可作为 Go 库嵌入到自定义应用中

与 Appwrite、Supabase 相比，PocketBase 最突出的特征不是“能力最多”，而是“形态更贴近底座层”。它更接近一个可嵌入、可加壳的本地后端内核，而不是一整套完整平台。

尤其关键的一点是：PocketBase 既可以作为现成服务使用，也可以作为 Go 库嵌入自定义逻辑。这使它非常适合承担本项目中的“底座 + 少量业务扩展”角色。

#### 局限

PocketBase 的弱点同样明显：

- 它不是完整平台，很多业务能力需要自己补
- 它没有成熟的前台展示层
- 它原生不等于直接满足分享访问、Hash URL、搜索高亮等业务规则
- README 还明确提示：在到达 `v1.0.0` 前不保证完全向后兼容

这意味着选择 PocketBase 的前提，不是把它当成成品系统，而是明确接受“它负责底座，我们负责业务壳”。

#### 判断

如果把“功能完备”作为唯一标准，PocketBase 不会赢；但如果把“是否适合作为本项目的本地轻量底座”作为标准，PocketBase 明显更贴近问题结构。

### 5.5 三者对比结论

综合以上比较，可以得到一个清晰判断。

#### Appwrite 与 Supabase 为什么没有成为首选

它们都很强，但主要问题不是“做不到”，而是“对于当前项目来说过于完整”。

- `Appwrite` 更像全栈应用平台，覆盖范围从认证、数据库到函数、消息和站点托管。
- `Supabase` 更像以 Postgres 为核心的完整开发平台栈，组件组合更多，平台属性更强。

对于当前这个单机边缘子系统，它们的能力边界都明显大于需求边界。若以它们为底座，系统很可能获得一套更重的平台，而不是一个更贴身的本地文件内容底座。

#### PocketBase 为什么成为当前底座首选

PocketBase 胜出的原因，不是因为它“功能最全”，而是因为它在本项目最关键的几个维度上更均衡：

- 复杂度更可控，契合当前 MiniPC 场景
- 自带文件、用户、API、后台等基础能力
- 既能独立跑，也能嵌入 Go 应用
- 更适合作为“被大项目调用的本地子模块”
- 为后续补业务壳留下了清晰边界

### 5.6 本章结论

重新比较 `PocketBase`、`Appwrite`、`Supabase` 之后，可以得出本报告自己的判断：

**如果目标是为“书童四九”构建一个本地、自托管、轻量、可扩展的文件与内容服务底座，那么当前最合适的开源底座是 `PocketBase`。**

这个结论成立的前提不是“PocketBase 功能最强”，而是：

- `Appwrite` 与 `Supabase` 都更偏完整平台，超出当前问题边界
- `PocketBase` 则最贴近“轻量底座”这一核心定位

但这一定义也同时意味着：

**选择 `PocketBase`，本质上是在选择“底座 + 业务壳”路线，而不是“一个开源项目直接交付全部功能”的路线。**

## 六、为什么不直接采用文件分享类项目作为主承载系统

在前文选择 `PocketBase` 作为底座之后，一个自然的问题是：既然候选池里已经有 `Gokapi`、`chibisafe`、`Pingvin Share`、`filebrowser` 这类非常接近“上传后生成可访问链接”的产品，为什么不直接采用它们作为本项目的主承载系统？

这里需要先把几个概念说清楚：

- `主承载系统`：指最终承担本项目核心数据模型、Open API、文件/内容存储与后续扩展责任的那一层系统。
- `底座层`：主承载系统在架构上的角色名称。本报告中，`主承载系统` 与 `底座层` 指向同一层。
- `参考设计层`：指不直接承担系统主干职责，但其局部设计值得吸收借鉴的开源项目。
- `自研业务壳`：指围绕本项目特有规则补上的那层很薄的自定义逻辑。

因此，这一章讨论的不是“文件分享类项目有没有价值”，而是：

**它们是否应该直接承担底座层，也就是成为本项目的主承载系统。**

本报告的判断是：它们非常适合进入参考设计层，但不适合作为主承载系统。

### 6.1 这些项目真正强在哪里

根据 README 信息，这类项目普遍在以下方面非常强：

- 文件上传与分享链接生成
- 访问控制、过期、密码、次数限制等分享治理
- 文件管理体验
- 大文件上传、分块传输、直链访问
- 外部用户访问文件的前台体验

例如：

- [Gokapi-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/Gokapi-README.md) 明确强调 `expiring file shares`、`REST API`、用户角色、文件请求、文件去重等能力。
- [chibisafe-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/chibisafe-README.md) 明确强调 `API key`、大文件分块上传、文件管理、标签、相册、片段直链等能力。
- [Pingvin-Share-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/Pingvin-Share-README.md) 强调带过期时间、密码、访问限制的分享链接模型。
- [filebrowser-README.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/开源项目调研/filebrowser-README.md) 则非常适合“工作目录浏览、上传、删除、预览、编辑”的本地文件管理视角。

这说明，文件分享类项目对本项目并不是“无关”，相反，它们恰好覆盖了本项目后半段链路里最容易做薄弱的部分。

### 6.2 这些项目为什么不适合作为主承载系统

问题在于，本项目并不是一个纯文件分享系统。它同时还要求：

- 接收 HTML 富文本内容并以 HTML 形式展示
- 统一管理文件与富文本两类内容
- 支持内容列表、搜索、详情
- 提供 Open API 供本地智能体调用
- 支持内容归属、后续 CRUD 与用户系统预留

也就是说，本项目需要的不只是“文件分享能力”，而是一个更基础的“内容与元数据底座”。

而文件分享类项目，大多是围绕“分享对象”设计的，不是围绕“统一内容模型”和“可扩展业务实体”设计的。若直接让它们承担底座层，通常会面临以下问题：

- 文件模型很强，但 HTML 富文本模型不自然
- 分享规则很强，但统一搜索与内容列表未必是核心模型
- 外部访问体验很强，但作为智能体写入底座不一定自然
- 很容易形成“分享工具很强，业务扩展很难”的局面

因此，文件分享类产品更适合作为能力参考源，而不是主承载系统。

### 6.3 各项目更适合参考什么

#### Gokapi

更适合参考：

- 分享链接生命周期
- 下载次数 / 时间驱动的过期机制
- 用户角色与分享访问限制
- 面向分享对象的轻量 REST API 设计

#### chibisafe

更适合参考：

- API Key 驱动的上传接入方式
- 大文件分块上传思路
- 文件管理、标签、直链和片段能力
- 上传后即分享的交互模型

#### filebrowser

更适合参考：

- 工作目录与本地文件树的映射关系
- 文件浏览、预览、编辑、删除这类管理视角
- 目录型文件管理 UI 的组织方式

#### Pingvin Share

更适合参考：

- 分享链接的配置项设计
- 密码、访问限制、过期时间等分享治理语义

但必须明确一点：README 已说明该项目处于 `Archived` 状态，因此它只能作为设计参考，不适合作为长期依赖的主选型。

### 6.4 本章结论

文件分享类项目并没有被排除在外；它们只是从“主承载系统候选”转变为“局部能力参考源”。

因此，本项目的更合理路线不是：

- 直接选一个文件分享产品承担底座层

而是：

- 选择一个轻量后端作为底座
- 再从文件分享类项目中吸收分享治理、上传体验和文件管理设计

## 七、推荐架构：一个底座 + 多个参考 + 薄业务壳

基于前文判断，本项目最合理的落地方式，不是“找一个项目一把梭”，而是采用明确分层的组合式方案。

### 7.1 底座层：选择一个轻量后端

底座层推荐使用 `PocketBase`，负责承担以下通用能力：

- 文件落盘与文件元数据保存
- HTML 内容与元数据保存
- 基础用户与认证能力
- 基础管理后台
- 面向智能体的基础 API 能力

选择它的理由已经在第五章给出：它不是最重的全栈平台，而是当前最贴近本项目问题结构的本地轻量底座。

### 7.2 参考设计层：从不同开源项目吸收局部能力

本项目不应把所有非底座能力都从零开始摸索。更现实的做法，是按能力拆解，借鉴不同项目已经验证过的设计。

建议参考关系如下：

- `分享链接与过期/撤销模型`：参考 `Gokapi`、`Pingvin Share`
- `上传入口、API Key、文件管理与片段能力`：参考 `chibisafe`
- `本地目录浏览与管理界面思路`：参考 `filebrowser`
- `内容建模和后台字段组织方式`：参考 `Directus`、`Strapi`
- `1Panel 运维友好性视角`：可参考 `Halo` 的部署经验，但不采用其系统模型

这种方式的重点，不是复用它们的全部实现，而是复用它们已经成熟的设计语言和产品拆法。

### 7.3 自研薄业务壳：只补本项目真正特有的规则

在底座层和参考设计层之外，仍然需要一层很薄的自研业务壳。这一层不追求大而全，只负责本项目特有的业务规则。

建议将其职责限定为：

- `API Key -> 业务用户` 的映射
- 内容归属校验
- 用户内容列表、详情、搜索接口
- `Hash URL` 路由生成与解析
- 分享链接生成、撤销和访问校验
- HTML 渲染模板与 XSS 清洗
- 文件流式下载转发
- 后续生命周期策略挂接点

这样做的好处是，底座层保持稳定，业务层保持轻量，分享与管理体验又可以借鉴成熟项目，而不是全部重新发明。

### 7.4 为什么这种组合比“全自研”或“全套现成”更合理

#### 比全自研更合理

因为底座、上传、分享、文件管理这些基础问题，开源社区已经有大量成熟经验，不需要重新造轮子。

#### 比全套现成更合理

因为本项目的业务规则组合比较特殊：

- 既有文件，又有 HTML 富文本
- 既有智能体写入，又有外部 HTTP 查看
- 既有本地落盘，又有分享访问治理
- 既要轻量，又要保留后续用户系统接口

这类组合通常没有一个现成产品能完全自然覆盖。硬套某个产品，往往会在后续演进阶段付出更大改造代价。

### 7.5 本章结论

本项目最合理的目标架构应明确写成：

**`PocketBase 作为底座 + 参考文件分享类与内容平台类项目的局部设计 + 自研一层很薄的业务壳`**

这不是折中方案，而是更符合当前问题结构的主方案。

## 八、能力拆解：哪些部分参考哪些项目

为了避免“调研了很多项目，最后只留下一个名字”，这一章将最终方案拆回能力视角，明确每类能力最值得参考哪些项目。

| 能力模块 | 主参考项目 | 参考理由 |
|---|---|---|
| 文件与元数据底座 | `PocketBase` | 轻量、内置文件与用户管理、适合本地底座 |
| Open API 与基础用户能力 | `PocketBase`、`Appwrite` | 前者更轻，后者可作为更完整 API 平台的对照参考 |
| 富文本内容建模 | `Directus`、`Strapi` | 更擅长内容字段组织、模型设计与后台管理思路 |
| 分享链接生命周期 | `Gokapi`、`Pingvin Share` | 过期、限制、撤销等分享治理模型更成熟 |
| 上传体验与 API Key 入口 | `chibisafe` | 上传即分享、API Key、分块上传、文件管理体验清晰 |
| 本地目录管理视角 | `filebrowser` | 目录树、预览、编辑、删除等工作目录交互成熟 |
| 运维接入与 1Panel 友好性 | `Halo` | 虽不适合作主承载系统，但其 1Panel 部署友好性值得参考 |

从这个映射可以看出，最终路线不是“只选一个项目”，而是“底座有一个主选型，其余项目各自贡献特定能力的设计参考”。

## 九、实施建议与阶段划分

为了避免一开始就把所有规则压到同一轮实现，本项目建议按阶段推进。

### 9.1 MVP 阶段

MVP 阶段应先打通最小闭环：

- `PocketBase` 落地为本地底座
- 完成文件与 HTML 内容接入
- 完成基础元数据存储
- 提供列表、搜索、详情三个主流程
- 提供基础下载 / 查看链路
- 提供最基础的分享链接能力

这个阶段的目标不是把所有安全治理和生命周期规则一次做满，而是先把“本地写入 -> 对外访问”的主链路稳定跑通。

### 9.2 增强阶段

在 MVP 稳定之后，再补齐更偏业务治理的能力：

- 完整的 `Hash URL` 路由策略
- 分享链接撤销、过期、一次性访问等增强规则
- 生命周期管理
- 访问次数统计
- 大文件上传/下载优化
- 更完善的搜索表现，例如高亮和更细的过滤条件

这种分阶段策略，与原始需求中对优先级的描述是一致的，也能降低第一轮方案落地的复杂度。

## 十、最终结论

基于本轮扩展后的候选池、原始需求以及业务规则，可以得出以下结论。

### 10.1 路线层结论

本项目的主路线，应优先选择 `轻量后端 / BaaS 底座`，而不是：

- 建站 / 笔记类产品
- 纯文件分享类产品
- 更重型的内容平台作为第一优先

原因不是这些类别没有价值，而是它们要么偏前台成品、要么偏单一分享能力、要么对当前边缘场景明显偏重。

### 10.2 底座层结论

在 `PocketBase`、`Appwrite`、`Supabase` 三者中，当前最适合作为本项目底座层的是 `PocketBase`。

这个结论不是沿用旧报告，而是重新比较后得出的判断：

- `Appwrite` 与 `Supabase` 都更像完整平台，能力更大，但边界也更重。
- `PocketBase` 更贴近“单机、本地、轻量、可嵌入”的问题结构。
- 因此它更适合作为“书童四九”的本地文件与内容底座。

### 10.3 架构层结论

本项目不应尝试找到一个“原生就完全满足全部需求”的开源产品。更合理的架构策略是：

- 选择一个轻量底座：`PocketBase`
- 参考文件分享类项目的成熟设计：`Gokapi`、`chibisafe`、`filebrowser`、`Pingvin Share`
- 再自研一层很薄的业务壳，用于承接本项目特有的用户归属、Hash URL、分享访问与内容查询规则

### 10.4 最终推荐表述

因此，本报告的最终推荐不应写成“直接选某某现成开源项目交付”，而应写成：

**推荐采用 `PocketBase` 作为本地轻量底座，并参考 `Gokapi`、`chibisafe`、`filebrowser` 等项目的局部设计，补上一层很薄的自研业务壳，以满足“书童四九”文件管理与发布服务的业务需求。**

## 附录 B：本次选型的判断顺序

为了避免“先有结论、再找证据”，本报告采用的是如下判断顺序：

1. 先根据 [用户原始需求.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/用户原始需求.md) 与 [业务规则备忘录.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/20-planning/业务规则备忘录.md) 抽取决策原则。
2. 再将候选项目按产品类型分层，避免把不同赛道项目强行放进同一个比较表。
3. 然后先判断哪一类产品更适合作为主路线。
4. 在主路线收敛到 `轻量后端 / BaaS 底座` 后，再重新比较 `PocketBase`、`Appwrite`、`Supabase`。
5. 最后再决定哪些能力参考文件分享类项目，哪些能力保留给自研业务壳。

这个顺序的意义在于：

- `PocketBase` 不是预设结论，而是赛道收敛后得出的底座结论。
- 文件分享类项目不是“失败者”，而是被重新定位为局部能力参考源。
