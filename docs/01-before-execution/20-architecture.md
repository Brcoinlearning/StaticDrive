# 架构设计文档

## 1. 本轮架构目标

为“书童四九”本地文件管理与发布服务定义一套适合 MiniPC 单机部署的 MVP 架构，打通以下主链路：

- 智能体通过 Open API 写入文件或 HTML 富文本。
- 系统完成内容落盘、元数据保存和内容归属绑定。
- 发布者可查看自己的内容列表、搜索结果和详情。
- 外部用户可通过分享链接访问内容。

## 2. 本轮明确不采用的方向

- 不建设通用 CMS 或建站平台架构。
- 不将文件分享产品作为系统主承载层。
- 不把 PocketBase 之外的重平台组件引入 MVP 主链路。
- 不自研替代 PocketBase 的底层数据库、文件管理或后台底座。
- 不在本轮引入独立全文检索、消息队列或对象存储系统。
- 不在本轮实现复杂生命周期调度、统计分析和多租户权限体系。

## 3. 结构边界

本系统采用三层结构：`PocketBase 底座 + 业务壳服务 + 对外访问层`。

其中有两条强约束：

- `PocketBase` 是明确选定的开源底座，后续实现必须建立在该底座之上，不允许在 MVP 阶段手工重写一套等价底座来替代它。
- 文件分享、上传体验和目录管理能力允许参考 `Gokapi`、`chibisafe`、`filebrowser`、`Pingvin Share` 等开源项目的设计，但这些项目仅作为局部能力参考，不替代主承载层。

### 3.1 PocketBase 底座层

职责：

- 以开源底座的形式被实际安装或引入到当前项目运行环境中。
- 保存用户、内容、分享链接等结构化记录。
- 管理底层文件存储。
- 提供基础数据库、文件与管理能力。

边界：

- 不直接承担业务归属规则、分享访问校验和 hash URL 解析规则。
- 不直接作为外部用户访问前台。

### 3.2 业务壳服务层

职责：

- 提供面向“书童四九”的 Open API。
- 承担 API Key 到业务用户的映射。
- 维护统一内容模型，屏蔽文件与 HTML 富文本的底层差异。
- 生成内容 hash、分享 hash，并执行业务权限校验。
- 协调 PocketBase 数据记录与本地文件落盘的一致性。

边界：

- 不负责复杂站点展示。
- 不直接承担最终 HTML 页面渲染样式体系，只提供内容访问能力和查询接口。

### 3.3 对外访问层

职责：

- 提供只读网页入口。
- 展示内容列表、搜索结果和详情页。
- 根据内容类型决定 HTML 渲染或文件下载。
- 消费业务壳暴露的前台访问接口。

边界：

- 不直接访问 PocketBase 内部数据结构。
- 不承担后台管理与写入职责。

## 4. 模块职责

### 4.1 身份与归属模块

- 验证 API Key。
- 解析并绑定业务用户。
- 为所有内容写入用户归属信息。

### 4.2 内容接入模块

- 接收文件上传请求。
- 接收 HTML 富文本保存请求。
- 规范化输入参数和元数据。

### 4.3 内容存储模块

- 执行文件落盘。
- 将内容元数据写入 PocketBase。
- 维护内容类型、标题、原始文件名、hash、存储路径等字段。

### 4.4 分享访问模块

- 生成内容访问 hash。
- 生成分享链接并支持撤销。
- 校验分享链接有效性。

### 4.5 内容查询模块

- 提供内容列表、分页、排序。
- 提供按文件名和标题的搜索能力。
- 提供内容详情查询。

### 4.6 前台展示模块

- 渲染列表页。
- 渲染搜索页。
- 渲染详情页或触发文件下载。

## 5. 核心数据模型

### 5.1 User

- `id`
- `api_key`
- `display_name`
- `created_at`

### 5.2 Content

- `id`
- `owner_user_id`
- `type`：`file | rich_text`
- `title`
- `original_filename`
- `content_hash`
- `storage_path`
- `mime_type`
- `file_size`
- `html_content`
- `is_shared`
- `created_at`
- `updated_at`

### 5.3 ShareLink

- `id`
- `content_id`
- `share_hash`
- `is_revoked`
- `created_at`

## 6. 关键流程

### 6.1 文件写入流程

1. 智能体携带 API Key 调用上传接口。
2. 业务壳验证 API Key 并确定用户归属。
3. 内容接入模块规范化文件信息。
4. 内容存储模块生成 `content_hash`，执行文件落盘并写入 PocketBase 记录。
5. 分享访问模块生成默认访问地址或分享链接。
6. 返回内容 ID、hash 和访问地址。

### 6.2 富文本写入流程

1. 智能体携带 API Key 调用富文本保存接口。
2. 业务壳验证身份并构造统一内容对象。
3. 内容存储模块保存 HTML 内容和元数据。
4. 分享访问模块生成访问地址。
5. 返回内容 ID、hash 和访问地址。

### 6.3 发布者查询流程

1. 发布者通过受保护接口请求列表、搜索或详情。
2. 业务壳按 `owner_user_id` 过滤可见范围。
3. 返回结构化内容结果。

### 6.4 外部访问流程

1. 外部用户访问分享 URL 或内容 URL。
2. 对外访问层将 hash 提交给业务壳解析。
3. 业务壳验证内容或分享链接有效性。
4. 若为 HTML 内容，则返回可渲染内容；若为文件，则返回下载或预览响应。

## 7. 依赖方向

- 对外访问层 依赖 业务壳服务层。
- 业务壳服务层 依赖 PocketBase 底座层。
- 业务壳服务层 可依赖本地文件系统。
- PocketBase 底座层 不反向依赖业务壳或对外访问层。
- 对外访问层 禁止直接依赖 PocketBase。

## 8. 明确禁止的依赖关系

- 前台页面直接读取 PocketBase 数据库或内部表结构。
- PocketBase hook 中塞入完整业务流程，导致业务规则分散失控。
- 文件访问 URL 直接暴露原始文件路径或原始文件名。
- 内容类型分叉成两套完全独立的 API 和页面体系。
- 为了“开发更自由”而绕开 `PocketBase`，临时手搓一套自定义存储底座。
- 把参考项目中的产品边界整块搬进来，导致主承载层从 `PocketBase` 漂移为别的系统。

## 9. 模块衔接方式

### 9.1 PocketBase 与业务壳

- PocketBase 负责基础数据持久化。
- 业务壳通过 SDK 或 HTTP API 访问 PocketBase。
- 所有领域规则优先收敛在业务壳，而不是写散在底座配置中。
- 在进入业务壳主要开发前，PocketBase 必须已经完成安装或引入、基础模型初始化和最小连通性验证。

### 9.2 业务壳与本地文件系统

- 文件类型内容落盘到工作目录。
- 业务壳维护数据库记录与物理文件路径的映射关系。
- HTML 富文本优先以记录内容形式保存，必要时再补渲染模板。

### 9.3 对外访问层与业务壳

- 对外访问层不自行计算权限，只调用业务壳提供的已校验查询接口。
- 页面层专注展示，不复写权限和 hash 规则。

## 10. 运行与部署边界

- 部署目标为单机 MiniPC。
- PocketBase、业务壳和网页层可作为同机多个进程或容器运行。
- 不要求独立微服务拆分。
- 允许后续在 1Panel 中以多应用方式编排，但 MVP 不依赖复杂编排能力。

## 11. 风险点与控制策略

### 风险 1：文件落盘与元数据写入不一致

控制策略：

- 采用清晰的写入顺序和失败补偿。
- 写入失败时回滚记录或删除残留文件。

### 风险 2：业务规则过度下沉到 PocketBase

控制策略：

- 将 API Key 映射、归属校验、分享校验、hash 解析统一放在业务壳。
- 仅将底层存储逻辑留在 PocketBase。

### 风险 3：文件与 HTML 富文本分裂为两套实现

控制策略：

- 统一 `Content` 模型。
- 统一列表、搜索、详情接口，只在渲染末端按类型分支。

### 风险 4：前台绕过业务壳直接读底层

控制策略：

- 明确禁止对外访问层直连 PocketBase。
- 所有页面数据都经由业务壳接口返回。

## 12. 当前建议的实现落点

- `PocketBase`：底层数据与文件能力。
- `业务壳服务`：Go 实现更自然，可与 PocketBase 生态和单机部署目标保持一致。
- `网页层`：可采用服务端渲染或轻量前端应用，但必须保持依赖方向正确。

## 13. 实现期强约束

- 实现期必须复用 `PocketBase` 作为底座，不允许把“先手搓、后续再替换”当作临时策略。
- 实现期对 `Gokapi`、`chibisafe`、`filebrowser`、`Pingvin Share` 的使用方式仅限于借鉴分享治理、上传体验、目录管理等局部设计，不得把它们升级成新的主承载层。
- 业务壳负责补业务规则，不负责重造底座。
- 如果后续 agent 提出完全脱离 `PocketBase` 的新路线，应视为架构偏离，必须先回到文档阶段重新决策，不能直接编码。
