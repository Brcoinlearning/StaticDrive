# Phase 2 / Phase 3 Scope

本文档在重新评估 MVP 缺口后，定义后续 `Phase 2` 与 `Phase 3` 的边界。

## 1. 范围前提

本文件建立在以下前提上：

- 当前系统主链路已经基本打通。
- 但按原始需求重定义后的真正 MVP，仍存在若干未完全闭合的缺口。
- 因此后续阶段工作不能直接从“完整 MVP 已结束”这个假设出发，而应先识别哪些工作仍属于 MVP 收尾。

参考文档：

- [mvp-gap-reassessment.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/archive/phases/P2-Stabilization/01-before-execution/mvp-gap-reassessment.md)

## 2. 阶段边界重定义

### 2.1 MVP 收尾缺口

这部分工作虽然发生在 MVP 之后，但性质上仍然属于 MVP 完整性补足，而不是新阶段增强。

建议纳入：

- 文件原样下载一致性验证
- 真实文件类型交付验证
- 若确认“外部用户”=公开访客，则补公开列表/搜索/详情流

### 2.2 Phase 2

Phase 2 的目标应定义为：把当前系统从“可演示、可验证”推进到“稳定、可持续维护、适合长期运行”。

建议包含：

- 启动前自检
- 配置校验
- PocketBase 集合状态检查
- 更清晰的错误提示与日志
- 冷启动验证
- 部署与运维补强
- 备份恢复说明
- 更贴近真实环境的集成验证

不建议在 Phase 2 中默认包含：

- 完整 owner 产品化前端
- 大规模 UI 重做
- 全量浏览器 E2E 体系

### 2.3 Phase 3

Phase 3 的目标应定义为：把系统从“稳定可维护”推进到“更完整的产品化体验”。

建议包含：

- 更完整的 owner 前端
- 更自然的 owner 鉴权方式
- 内容更新能力
- 批量操作
- 搜索增强
- 更成熟的前端交互与状态管理
- 浏览器级 E2E 测试

## 3. 建议执行顺序

1. 先补足仍属于 MVP 的缺口
2. 再进入 Phase 2 稳定化与运维化
3. 最后进入 Phase 3 产品化建设

## 4. 当前共识结论

- 当前不应简单把所有未完成项都塞进 Phase 2 或 Phase 3。
- 需要先把“仍属于 MVP 的缺口”与“真正的后续阶段工作”分开。
- Phase 2 与 Phase 3 的 architecture / tasks 应建立在这个边界共识之上。
