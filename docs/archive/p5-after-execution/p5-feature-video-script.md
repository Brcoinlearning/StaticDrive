## 口播脚本

这次我要讲的新功能，基本上是围绕最开始那份原始需求来实现的。

原始需求分成两阶段。第一阶段主要是三件事：第一，新增字符串的写和查接口；第二，新增应用对字符串，也就是富文本内容的直接显示能力；第三，改进列表页面，让字符串内容能像正式资源一样被查看和管理。第二阶段，是在这个基础上再增加资源访问密码能力。

所以这次的技术实现，也不是零散加几个点，而是围绕这四件事，把字符串内容正式接入到了系统主链路里。

### 第一部分：先讲新增 API

这里我只讲这次新功能真正新增的 API，不把系统里原来就有的旧接口一起混进来。

如果按阶段来讲，这次新增的 API 主要是四个。

第一阶段，围绕“新增字符串的写、查接口”，新增了：

- `POST /api/write/content`
- `GET /api/query/content/:contentId`

其中，`POST /api/write/content` 负责把字符串内容正式写进系统。

它支持的核心参数包括：

- `title`
- `body`
- `bodyFormat`

其中 `bodyFormat` 目前主要支持：

- `html`
- `markdown`

这个设计背后的原理是，系统不能只知道“你传了一段文本”，还要知道“这段文本应该按什么格式理解”。

这样实现出来的能力是：

- 如果传入的是 `html`，系统就把它当成 HTML 正文处理
- 如果传入的是 `markdown`，系统就会先做 Markdown 渲染，再进入展示链路

而 `GET /api/query/content/:contentId` 负责把这条字符串内容按统一内容对象语义查出来。

它返回的重点字段包括：

- `body`
- `bodyFormat`
- `renderedBodyHtml`
- `htmlContent`

它们分别对应：

- 原始正文
- 正文格式
- 渲染后的最终 HTML
- 兼容展示使用的 HTML 内容

也就是说，第一阶段新增的不是孤立的“写入接口”，而是“写进去”和“按统一语义查出来”这一对新接口。

第二阶段，围绕“支持资源访问密码”，新增了：

- `POST /api/public/content/:contentHash/password`
- `POST /api/public/share/:shareHash/password`

这两个接口对应的是密码访问闭环。

它们的作用不是返回正文，而是先完成密码验证。验证通过以后，浏览器或调用方才拿到继续访问这条内容的短期许可。

所以这一段在视频里更准确的说法应该是：

**这次新功能真正新增的 API，第一阶段是字符串写入和字符串查询接口，第二阶段是 public 和 share 两条外部访问链路上的密码校验接口。**

### 第二部分：字符串内容为什么能成为正式资源

新增 API 只是入口，真正更重要的是，系统内部怎么接住这条新链路。

这里我们用了一个统一内容对象模型。

它的原理是，不管输入来源是：

- 文件上传
- HTML 正文
- Markdown 正文

进入系统后，都尽量收敛成同一条 `contents` 记录来管理。

这样实现了一个非常关键的效果，就是字符串内容不会被当成“临时文本”，而会被当成正式资源。

正式资源意味着什么？意味着它后面会拥有和别的内容一样的管理能力，比如：

- 有 owner
- 有内容标识
- 能进列表
- 能看详情
- 能 public 展示
- 能 share 展示
- 能挂访问控制

这也是为什么这次改动看起来像是在加字符串支持，但实际上做的是内容模型收敛。

### 第三部分：字符串内容是怎么保存的

这次在正文存储上，我们用了“原始正文 + 最终展示 HTML”的双轨存储机制。

它的原理是，把作者写的原稿，和系统最终展示给页面看的结果，拆成两份保存。

具体来说：

- `body_source` 保存原始正文
- `body_format` 保存原始正文格式
- `html_content` 保存最终展示 HTML

这样做实现了两个结果。

第一个结果是，owner 后续还能继续编辑原稿。

比如用户最开始写的是 Markdown，如果系统只保留渲染后的 HTML，那后面再编辑的时候，就已经失去 Markdown 原稿了。现在因为原始正文被单独保存下来了，所以 owner 还能继续按原来的格式去修改。

第二个结果是，owner 页面、public 页面、share 页面可以复用同一份最终 HTML，而不是每个页面各自再解析一次正文。

这样就减少了展示不一致的问题。

### 第四部分：新增了应用对字符串内容的直接显示能力

原始需求里提到，应用要支持对字符串，也就是富文本内容的直接显示。

这里的实现关键，不是简单把正文原样打印出来，而是先把它整理成可展示的 HTML，再进入页面层。

如果 `bodyFormat=html`，系统会直接把正文当成 HTML 处理。

如果 `bodyFormat=markdown`，系统会先在服务端把 Markdown 转成 HTML，再把最终结果提供给页面。

这一步我们用了服务端预渲染机制。

它的原理是，Markdown 不是等页面打开时再临时转换，而是在内容创建和更新的时候就先转换好。

这样实现的好处是：

- 页面层更轻
- owner/public/share 三条链路更一致
- 后续调试时也更容易定位问题是在写入阶段还是展示阶段

### 第五部分：Markdown 具体支持什么显示能力

如果视频里要专门讲 Markdown，这一段可以重点说，因为这不是“只支持几行纯文本”的能力，而是支持比较完整的结构化富文本。

当前项目里，Markdown 渲染大体支持这些能力：

- 标题
- 普通段落
- 粗体、斜体
- 行内代码
- 围栏代码块
- 无序列表
- 有序列表
- 嵌套列表
- 任务列表，也就是带勾选框的列表
- 引用块
- 表格
- 分隔线
- 行内链接
- 裸 URL 自动识别成链接
- 图片语法
- 行内数学公式
- 块级数学公式

这背后的原理是，渲染器不是做一次简单字符串替换，而是按两层来处理。

第一层是块级解析，负责识别：

- 标题
- 列表
- 表格
- 引用
- 代码块

第二层是行内解析，负责识别：

- 链接
- 图片
- 粗体
- 斜体
- 行内代码
- 行内公式

这样实现出来的效果，就是 Markdown 不只是“能展示”，而是能比较稳定地把结构也展示出来。

### 第六部分：Markdown 里的图片是怎么处理的

这部分也可以单独讲，因为很多人会关心图片是不是也接进来了。

当前 Markdown 里的图片，处理方式是：

- 识别 Markdown 图片语法
- 把它转换成 HTML 的 `<img>` 标签
- 页面最终直接展示这个图片标签

比如：

```md
![架构图](https://example.com/demo.png "示意图")
```

最后会被转换成类似：

```html
<img src="https://example.com/demo.png" alt="架构图" title="示意图">
```

这里要特别说明的是，当前系统不会把 Markdown 里的图片自动下载到本地，也不会把它自动变成系统里的 file 资源对象。

它现在的实现更像是：

**正文里允许嵌入一个外部图片地址，并在最终 HTML 里把它显示出来。**

这样做实现起来更直接，但也意味着图片可用性依赖外部地址本身是否还能访问。

### 第七部分：渲染安全到底是什么意思

“渲染安全”这个词如果直接讲，确实会有点抽象。更好讲的方式是，你可以把它理解成：

**系统在把用户写的正文变成网页内容时，要防止正文里混进危险代码，或者把页面结构搞乱。**

比如，用户输入的内容里如果出现这类东西：

```html
<script>alert(1)</script>
```

如果系统不做处理，最糟糕的情况是，这段东西可能被浏览器当成真的脚本执行。

所以当前实现里，第一步会先做 HTML 转义。它的原理是，先把这些特殊符号变成普通文本，而不是直接当成 HTML 标签。

你可以理解成，系统先把危险尖括号“去武器化”，让它们只能显示，不能执行。

第二步，在 Markdown 渲染过程中，系统会把已经生成好的安全 HTML 片段先临时保护起来。

这样做的原因是，Markdown 解析往往要做很多轮替换。比如先处理图片，再处理链接，再处理粗体、斜体、代码。如果没有保护机制，前面刚生成好的 `<img>`、`<a>`、`<code>`，后面可能又被下一轮替换误伤，最后把页面结构弄乱。

所以你可以把这套机制讲成两句话：

- 第一，防止用户输入里的危险内容直接变成可执行网页代码
- 第二，防止 Markdown 多轮解析过程中把已经生成好的 HTML 结构再破坏掉

这样“渲染安全”就不只是一个技术术语，而是两个很具体的问题：

- 安不安全
- 稳不稳定

### 第八部分：列表页面为什么也要改

原始需求里还提到要改进列表页面，增加列表视图，并优化显示项内容，比如：

- 来自用户
- 时间
- 标题
- 主要内容

这一步的关键不只是 UI 变化，而是列表页开始接受“字符串内容也是正式内容对象”这个事实。

也就是说，列表页不再只适合文件记录，也能比较自然地展示：

- 这是谁写的
- 什么时候写的
- 标题是什么
- 正文概要是什么

这样实现了原始需求里说的“字符串资源可直接查看和管理”的感觉。

### 第九部分：浏览器里也补了一个直接写入入口

除了 API，这次还新增了 `/web/write` 页面。

它的意义不是另起一套系统，而是给同一条字符串写入主链路加了一个浏览器入口。

它的原理是：

- `GET /web/write` 渲染写入表单
- `POST /web/write` 把表单内容交给同一个内容服务创建记录

这样实现了一个非常适合演示和验收的能力，就是除了 API 和 Agent 写入之外，owner 也可以直接在前端模拟字符串写入。

### 第十部分：第二阶段的密码访问是怎么实现的

原始需求的第二阶段，是支持资源访问密码，类似网盘分享时需要密码访问。

这里我们用了内容级访问控制机制。

它的原理是，把访问模式直接挂在 `contents` 记录上，而不是挂在某个页面上。

具体新增了三个关键字段：

- `access_mode`
- `access_password_hash`
- `access_hint`

它们分别负责：

- 这条内容是公开访问还是密码访问
- 密码的 hash 存储
- 给访客的提示语

这样实现的关键结果是：

- public 入口和 share 入口，不会各自维护一套密码逻辑
- 它们最终都会回到“这条内容自己的访问规则”上来判断

也就是说，密码保护不是页面装饰，而是内容对象的一部分。

### 第十一部分：密码为什么不是只加一个输入框

密码能力真正落地时，我们用了密码哈希校验机制和短期访问态机制。

密码哈希校验机制的原理是：

- 系统不保存明文密码
- 只保存 hash
- 访客输入后，由服务端校验输入值和 hash 是否匹配

这样实现了密码保护，同时避免把明文密码直接留在数据库里。

短期访问态机制的原理是：

- 用户第一次输入正确密码后
- 服务端给当前浏览器一张短时间有效的访问许可

这样实现了用户体验上的闭环，也就是：

- 刚输完密码后刷新页面，不需要立刻再输一次
- 但这个权限又不是永久的，也不是 owner 登录态

### 第十二部分：错误密码为什么还能重试

这次密码能力还有一个很重要但容易被忽略的点，就是错误密码可重试。

这里我们用了可恢复失败状态设计。

它的原理是，把访问过程分成三种状态：

1. 未验证
2. 验证失败
3. 验证成功

这样实现的结果是，用户输错密码时，不会被直接打到一个死页面，而是仍然留在密码页，并且看到密码框下方的红字错误提示，然后继续再试。

这一步对用户体验很关键，因为它保证的是一个真正可用的访问闭环，而不是“理论上支持密码”。

## 结尾版总结

如果用一句话总结这次功能，可以这样说：

这次我们围绕原始需求，新增了字符串内容的写入和查询 API，把字符串内容接入了统一 `contents` 内容模型，用服务端预渲染机制实现了 HTML 和 Markdown 的直接显示能力，补齐了列表页和浏览器写入入口，最后再通过内容级访问控制、密码哈希校验和短期访问态，把资源密码访问这条链路也闭合了。

如果要再压缩成更技术的一句话，可以讲：

我们用了统一内容对象模型、双轨正文存储、服务端 Markdown 预渲染和内容级密码访问机制，把字符串内容从一次输入，升级成了一个可写、可查、可展示、可分享、可控访问的正式资源对象。
