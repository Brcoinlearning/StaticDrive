## 1. 这轮实现的核心目标

这轮本质上是在做一件事：

**把“字符串内容”提升成系统的一等公民。**

以前系统更偏两种输入：

1. 文件上传
2. 原始 HTML 写入

现在新增的是第三种，而且是很重要的一种：

3. **正文字符串写入，尤其是 Markdown 字符串写入**

这意味着系统内部不能再把“字符串正文”当成某个临时字段，而必须把它纳入完整的内容生命周期：

- 创建
- 查询
- 更新
- owner 展示
- public 展示
- share 展示
- 访问控制

所以实现上不是只加一个接口，而是把整条链路都补齐。

## 2. 先统一内容模型

这是最关键的第一步。

如果系统内部还把数据分成：

- 文件内容一套逻辑
- HTML 一套逻辑
- Markdown 再一套逻辑

后面你每做一个页面、每做一个分享、每加一个权限规则，都会分叉。

所以这一轮的真正做法是先统一“内容对象”的概念。

也就是说，不管外面传进来的是什么：

- 文件
- HTML 正文
- Markdown 正文

在系统内部，最后都尽量落成一条统一的 `content` 记录。

这条记录至少要能回答这些问题：

- 它是谁的内容
- 它的标题是什么
- 它属于什么类型
- 它有没有公开
- 它对外访问地址是什么
- 它现在的访问模式是 `public` 还是 `password`

这一层统一以后，后面的页面和 API 才能围绕“内容对象”工作，而不是围绕“输入来源”工作。

### 你可以这样理解

以前更像：

- 文件是一类系统对象
- HTML 是另一类系统对象

现在变成：

- 文件和正文，都只是“内容对象”的不同来源或不同形态

这是架构上最重要的收敛。

## 3. 对富文本内容，系统同时保存“原文”和“最终展示结果”

这一步决定了为什么这轮能稳定。

对于字符串内容，尤其是 Markdown，系统没有只存一份东西，而是存了两类信息：

1. **原始正文**
2. **最终展示 HTML**

这两个字段的职责不同。

### 原始正文是干什么的

原始正文负责：

- 保留用户最初提交的内容
- 让 owner 以后还能继续编辑
- 保证你不会因为渲染过程而丢失原稿

比如你写入的是 Markdown：

```md
# 标题

- 第一项
- 第二项
```

系统不会把这段原文丢掉。

因为如果丢掉了，后面 owner 再打开编辑页时，你只能拿到渲染后的 HTML，就没法顺畅地继续按 Markdown 语义编辑了。

### 最终展示 HTML 是干什么的

最终 HTML 负责：

- owner 详情页预览
- public 页面展示
- share 页面展示

页面层拿到它以后，就不用再临时理解 Markdown 语法了。

这很关键，因为页面层如果每次打开都临时做 Markdown 转换，会带来几个问题：

- 每个页面都可能实现一套自己的转换逻辑
- owner/public/share 结果不容易完全一致
- 运行时出错会变多
- 调试更麻烦

所以这轮采用的是更稳定的方式：

**正文在写入或更新时就转换好，页面只消费结果。**

## 4. Markdown 的转换发生在写入期和更新期，不发生在展示期

这是这轮架构边界里非常重要的一条规则。

### 为什么不在展示期转换

如果在页面打开时才转换 Markdown，会有几个坏处：

1. 每次访问都要额外做一次解析
2. public 和 share 很容易走成两套逻辑
3. 页面层承担太多格式处理责任
4. 同一条内容的展示结果可能随页面实现变化

### 现在的实现方式

系统现在的思路是：

- 创建内容时，如果 `bodyFormat=markdown`，先把 Markdown 转成 HTML
- 更新内容时，如果正文或格式变了，再重新生成 HTML
- 查询和页面层只拿最终结果展示

这样做的实际效果是：

- owner 详情页看到什么
- public 页看到什么
- share 页看到什么

它们理论上应该基于同一份最终内容结果。

### 这一步的价值

这一步实际上是在把系统拆成两个职责层：

1. **内容加工层**
   负责把用户输入整理成系统最终可展示的结果

2. **内容展示层**
   只负责把结果安全、稳定地展示出来

这个边界立住以后，系统就不容易乱。

## 5. `/web/write` 不是单独的一套系统，而是同一条写入链路的浏览器入口

这一点很容易误解。

`/web/write` 页面看起来像是新功能，但它本质上不是一套独立业务，而是把“字符串写入主链路”开放给浏览器使用。

### 它内部实际在做什么

浏览器访问 `/web/write` 时：

- `GET /web/write`
  渲染一个写入表单页

用户提交表单时：

- `POST /web/write`
  服务端读表单内容，再调用内容服务创建一条新的富文本内容

也就是说，这个页面本质上只是把原本 API 才能干的事，包装成了一个更适合人工验证的入口。

### 为什么这页重要

因为如果没有它，你只能证明：

- 接口可以写入字符串

但有了它，你还能证明：

- owner 浏览器会话体系可以正常工作
- 页面级表单提交能打通字符串写入
- 非 API 调用者也能直接验证这条能力

所以它在技术上是“写入主链路的前端入口”，不是额外并行的一套逻辑。

## 6. owner 页面这轮为什么要改

如果只是后端能写 Markdown，但 owner 页面还停留在旧心智，这条链路仍然不完整。

所以 owner 页面这轮至少做了两件事。

### 6.1 列表页接受“这是一条正式内容，而不一定是文件”

owner 列表现在不能再默认：

- 一条内容必须有原始文件名
- 没有文件就像半成品

它要能接受：

- 这是一条富文本内容
- 它的重点是标题、摘要、作者、创建时间
- 它同样是系统里的正式对象

这其实是页面层在跟着底层模型统一。

### 6.2 详情页右侧更新表单从“改标题”升级成“改内容语义”

这一轮以后，owner 详情页右侧更新表单不只是编辑一些表面字段，而是真正能改：

- `title`
- `body`
- `bodyFormat`
- `accessMode`
- `accessPassword`
- `accessHint`

这说明更新接口也已经被升级成内容对象语义了。

不是只改“展示文本”，而是在改：

- 原文是什么
- 原文格式是什么
- 最终访问方式是什么

这一步很重要，因为如果创建支持 Markdown、更新却不支持，那整条链路还是断的。

## 7. public 和 share 页面这轮为什么会一起改

因为这轮不是在做“owner 侧单点增强”，而是在做“内容对象主链路增强”。

如果 owner 能写 Markdown，但 public 和 share 还按旧逻辑展示，会出现典型的不一致：

- owner 里看起来正常
- public 页展示错乱
- share 页又是另一种结果

所以这轮的关键不是“多改两个页面”，而是让它们尽量复用同一份最终渲染结果。

### public 页

public 页要做到：

- 根据内容 hash 找到这条内容
- 如果它允许公开访问，就返回最终展示 HTML
- 页面中展示的是渲染结果，不是原始 Markdown

### share 页

share 页要做到：

- 根据 share hash 找到分享记录
- 再找到对应内容
- 返回同一条内容的最终展示结果

所以从实现角度看，public 和 share 的差别更多在于“入口定位方式不同”，而不是正文处理方式不同。

一个按 content hash 找内容，一个按 share hash 找分享关系再找内容。

但最后展示的内容结果，应该尽量统一。

## 8. password 模式为什么不是简单加个输入框

这是这轮第二个关键点。

如果只是页面上多一个密码框，那它根本不算真正实现了访问控制。

真正的实现必须把“访问模式”收进内容对象本身。

也就是说，一条内容现在不仅有标题和正文，还多了访问控制属性，例如：

- `access_mode`
- `access_password_hash`
- `access_hint`

### 为什么要存 hash，不直接存明文密码

因为明文密码风险太高。

如果直接把密码原文存进去，一旦日志、数据库、调试过程有泄漏，风险很大。

所以正常实现会保存密码的安全表示，也就是 hash。

系统后续不是比对“原文密码是否完全等于数据库里的原文”，而是：

- 用户输入一个密码
- 系统用同样方法处理
- 看结果是否匹配已存的 hash

这样更安全。

### 为什么要有 access_hint

因为很多场景里，密码并不是完全私密的高安全凭据，而是“分享门槛”。

提示语能让 owner 提供一个轻量线索，比如：

- 项目代号
- 某个会议名
- 某段提示短语

这样访客体验会更自然。

## 9. public/share 密码校验为什么要做成同一套访问规则

这里很容易做错。

最常见的错误实现是：

- public 页加了密码校验
- share 页忘了加
- 或者 share 页只做了前端拦截，后端没真正卡住

这样就会出现绕过漏洞。

所以正确做法不是“给每个页面各自加一个密码输入框”，而是：

**先在内容访问层判断：这条内容现在是不是 password 模式。**

如果是，就不管你是从：

- public content 入口进来
- 还是从 share 入口进来

都必须经过同一套密码验证逻辑。

这样才能保证访问控制是“内容级规则”，不是“页面级装饰”。

## 10. 错密码可重试，背后其实是状态设计问题

你提过一个很重要的要求：

- 错密码不能直接卡死
- 要允许重试
- 还要有红字提示

这件事背后不是简单的前端提示问题，而是访问状态设计问题。

### 系统要区分三种状态

1. **未验证**
   只能看密码页

2. **验证失败**
   仍然留在密码页，但要带错误信息

3. **验证成功**
   给短期访问许可，允许进入正文页

如果系统没有把这三种状态分清楚，就会很容易做出这种糟糕体验：

- 错一次就跳 403
- 再也回不到输入页
- 用户不知道怎么继续

现在这一轮要求的实现，本质上是让失败也成为一个可继续恢复的状态，而不是终止状态。

## 11. 短期访问态是怎么实现思想上的闭环的

你可以把它理解成“验证成功后发一个短期通行证”。

如果没有这层设计，用户每刷新一次页面、每开一个页面，都要重新输密码，体验会非常差。

所以合理实现通常会做：

- 用户提交密码
- 服务端验证通过
- 服务端给浏览器一个短期有效的访问标记
- 在有效期内，这个浏览器继续访问这条内容时可以直接放行

这个访问标记通常不是永久凭据，也不是 owner 登录态，而是更小范围的、针对这条内容的短期访问许可。

这就是为什么它叫“短期访问态”。

## 12. 这轮为什么不只是加功能，而是在补契约

从工程角度看，这轮真正难的地方不是多写一个页面，而是要保证整套语义一致。

你会发现，这轮一直围绕几组字段在收敛：

- `body`
- `bodyFormat`
- `renderedBodyHtml`
- `htmlContent`
- `accessMode`
- `accessPassword`
- `accessHint`

这说明它做的不是零散功能，而是在收一个统一契约。

也就是说，系统要让每一层都对这些概念理解一致：

- 写入层怎么理解它们
- 更新层怎么理解它们
- 查询层怎么返回它们
- 页面层怎么展示它们
- 验收脚本怎么验证它们

只要这套契约不统一，表面上功能做出来了，后面还是会反复回退。

## 13. 这一轮最本质的技术升级是什么

如果用一句更偏技术、但仍然白话的话说：

**这轮把“字符串内容”从一个输入形式，升级成了一个有完整生命周期、有统一展示结果、有统一访问控制的正式内容对象。**

它真正做成的不是：

- 新增一个表单
- 新增一个接口
- 新增一个密码框

而是让下面这些东西真正互相接上了：

1. 字符串写入
2. Markdown 转换
3. owner 新建
4. owner 更新
5. owner 预览
6. public 展示
7. share 展示
8. public/password 访问控制
9. 错误密码重试
10. 短期访问态

这就是“怎么实现的”本质答案。

## 14. Markdown 字符串的完整链路

这里说的“完整链路”，不是只指“有个接口能写进去”，而是指这段 Markdown 从进入系统开始，到最后被 owner、public、share 使用，中间每一步都接上了。

### 14.1 输入进入系统

入口现在有两种：

1. API 入口：`POST /api/write/content`
2. 浏览器入口：`GET /web/write` 打开表单，`POST /web/write` 提交表单

这两个入口的区别只是“谁发起请求”不同：

- API 更像程序或 Agent 调用
- `/web/write` 更像人工在浏览器里模拟写入

但它们最后都要走到同一条“创建内容”的主链路。

### 14.2 系统先确认这是谁的内容

不管你是 API 调用还是浏览器表单提交，系统都要先知道：

- 当前是谁在创建内容
- 这条内容应该挂到哪个 owner 名下

API 场景通常是直接看请求头里的 `API Key`。

浏览器场景通常是先登录一次，后续通过 owner 会话识别当前用户。

这一层的目的很简单：**先确定 owner，再允许创建内容。**

### 14.3 系统把输入理解成“内容对象创建请求”

这时候，系统不再把它看成“只是来了一段字符串”，而是看成：**有人要创建一条新的内容对象。**

对于 Markdown 字符串来说，最关键的输入是：

- `title`
- `body`
- `bodyFormat=markdown`

有些场景下还可能带：

- `authorName`
- `createdAt`
- `accessMode`
- `accessPassword`
- `accessHint`

也就是说，这一步开始，系统已经不是在处理“纯文本”，而是在处理“带元信息的一条内容”。

### 14.4 系统校验这次写入是否合法

这一步会检查几件事：

1. 标题和正文是不是满足最小要求
2. `bodyFormat` 是不是系统支持的值
3. 当前 owner 是否有资格创建内容

对 Markdown 来说，重点校验的是：

- `body` 不能为空
- `bodyFormat` 必须是 `markdown` 或 `html`
- 如果传了不支持的格式，比如 `plain_text`，会直接拒绝

这一步的意义是：**在内容真正入库前，先把非法输入挡住。**

### 14.5 系统生成内容的统一标识

一旦校验通过，系统就会给这条内容生成一套统一身份信息，比如：

- `contentId`
- `contentHash`

这两个东西不是 Markdown 专属，也不是文件专属，而是所有内容对象都要有的统一身份标识。

它们后面会用于：

- owner 详情页定位
- public 页定位
- share 创建和访问
- 列表和查询返回

所以从这一步开始，Markdown 内容和 HTML 内容、文件内容，在“身份层”已经是统一的了。

### 14.6 系统保留原始 Markdown 正文

这一步非常关键。

如果用户传的是 Markdown，系统会把原始正文保留下来。也就是：

- 你写的 Markdown 原文不会丢
- 后面 owner 再编辑时，还能拿回原文继续改

这是因为 Markdown 不是最终展示格式，而是“作者输入格式”。

如果这一层不保留原文，后面只剩渲染好的 HTML，就会很难再按 Markdown 语义编辑。

所以这一步本质上是在保存：**作者的原稿。**

### 14.7 系统把 Markdown 转成最终展示 HTML

这是 Markdown 链路的核心加工步骤。

因为：

- Markdown 适合写
- 但不适合直接给访客展示

所以系统会在创建时把 Markdown 转成 HTML。

这一步的结果就是生成：

- `renderedBodyHtml`
- 同时兼容对外返回的 `htmlContent`

你可以理解成：

- `body` 是原稿
- `renderedBodyHtml` 是排版后的可读版本

### 14.8 系统把这条内容按统一内容对象写入存储层

此时系统已经有了：

- owner 身份
- 标题
- 原始正文 `body`
- 格式 `bodyFormat=markdown`
- 最终展示结果 `renderedBodyHtml`
- 内容身份 `contentId/contentHash`

然后它会把这些统一映射成底层存储字段，写进 `contents` 集合。

这一层你不用背字段名，但理解上可以记住：**系统不是只存“字符串”，而是把这条字符串整理成了一条标准内容记录。**

### 14.9 系统返回创建结果

创建成功后，系统会返回类似这些信息：

- `contentId`
- `contentHash`
- `accessUrl`
- 内容类型等元信息

这表示：

- 这条 Markdown 字符串已经不是一次临时请求
- 它已经变成系统正式内容
- 后续可以像别的内容一样被查询、查看、分享

### 14.10 owner 列表和详情页复用这条内容

写入成功以后，这条内容进入 owner 管理链路。

owner 列表页会把它当成普通内容展示：

- 标题
- 摘要
- 作者
- 创建时间

owner 详情页会做两件事：

1. 展示最终 HTML 结果
2. 保留原始 `body + bodyFormat` 给 owner 继续编辑

所以 owner 详情页实际上同时承担了：

- 预览最终效果
- 编辑原始正文

这也是为什么前面一定要同时保留原文和最终 HTML。

### 14.11 public 和 share 页面复用同一份最终展示结果

如果这条内容被公开或被创建分享链接，public/share 链路就会接手。

这里最关键的一点是：

- public 和 share 不会重新去理解 Markdown
- 它们直接拿已经生成好的最终 HTML

所以你在这些页面看到的内容，应该和 owner 详情页预览的是同一份结果。

这一步保证了：

- owner 看起来正常
- public 也正常
- share 也正常

不会三边各自渲染出不同效果。

### 14.12 如果访问模式是 password，还要经过密码校验

到这里为止，内容链路只是“内容可展示”。

如果这条内容的访问模式是：

- `public`

那 public/share 可以直接访问。

如果是：

- `password`

那 public/share 在进入正文前，必须先通过密码校验。

也就是说，Markdown 内容链路和访问控制链路在这里合流了。

这说明这条 Markdown 内容已经真正进入了系统主线，而不是停留在“只能被 owner 私下看见”的状态。

### 14.13 后续更新时，再重复“保留原文 + 重新生成最终 HTML”的过程

如果 owner 后面更新这条内容：

- 改正文
- 改格式
- 改访问模式

系统会重新走一遍关键加工步骤：

1. 保留新的原始正文
2. 按新的 `bodyFormat` 重新生成最终 HTML
3. 更新这条内容对象的存储记录

所以这条链路不是一次性写入逻辑，而是完整的生命周期逻辑。

### 14.14 一句话总结这条完整链路

Markdown 字符串链路其实就是：

**输入 Markdown -> 确认 owner -> 校验 -> 生成统一内容身份 -> 保留原文 -> 转成最终 HTML -> 写成标准内容对象 -> 被 owner/public/share 统一复用 -> 再受访问控制约束**

这才叫“完整链路”。

## 15. 之前的 HTML、文件、现在的字符串，什么叫统一

这个问题特别重要。

因为“统一”不是说三者长得完全一样，也不是说它们底层完全同构，而是说：

**系统在关键管理层面，把它们尽量当成同一种内容对象来对待。**

### 15.1 统一不是“输入长得一样”

文件输入一般是：

- 文件名
- MIME 类型
- base64 文件内容

HTML 输入一般是：

- `title`
- `htmlContent`

Markdown/字符串输入一般是：

- `title`
- `body`
- `bodyFormat`

你可以看到，输入本来就不一样。

所以“统一”绝对不是指：

- 三种请求参数完全一样

它们不会一样。

### 15.2 统一是“进入系统后被当成同一种内容记录”

真正的统一发生在系统内部。

虽然输入形式不同，但进入系统以后，它们都会被整理成一条内容记录。

也就是都进入 `contents` 这条主线，而不是每种输入各自一套平行世界。

所以统一指的是：

- 都有 owner
- 都有 contentId
- 都有 contentHash
- 都能查列表
- 都能看详情
- 都能创建分享
- 都能公开访问
- 都能受访问模式控制

这就是“统一内容对象”的真正含义。

### 15.3 统一是“页面和查询围绕同一套内容语义工作”

如果没有统一，页面层就会变成这样：

- 文件详情页一套逻辑
- HTML 详情页一套逻辑
- Markdown 再一套逻辑
- public 又各自分叉
- share 再各自分叉

这样系统会越来越难维护。

现在的统一做法是：

- 页面先拿到内容对象
- 再根据它的类型和字段决定展示方式
- 而不是先按输入来源分裂成三套页面模型

所以统一不是“没有差异”，而是：

**差异被限制在必要的位置，而不是扩散到整个系统。**

### 15.4 统一不是“底层存储完全相同”

这里要注意，统一不等于一模一样。

比如：

- 文件内容本体通常在磁盘
- 富文本正文通常直接存在内容记录里

这说明底层物理存储并不完全一样。

但它们仍然可以统一，因为统一关注的是更高一层：

- 如何被识别
- 如何被查询
- 如何被展示
- 如何被分享
- 如何被控制访问

所以你要区分两个概念：

1. **物理存储是否完全相同**
2. **业务对象是否统一**

这轮做成的是第 2 个，不是强行把第 1 个也完全抹平。

### 15.5 统一最明显的表现是什么

你可以用这几个现象判断“是不是统一了”。

#### 1. 它们都出现在 owner 列表里

文件也进列表，字符串内容也进列表。

#### 2. 它们都有详情页

不是只有文件能被当成“正式记录”。

#### 3. 它们都能被分享

如果字符串内容不能分享，那它就还不是完整内容对象。

#### 4. 它们都能被 public 访问或 password 保护

这说明访问控制作用在“内容对象”上，而不是只作用在某种输入类型上。

#### 5. 查询返回结构尽量统一

比如都围绕：

- 内容身份
- 标题
- 展示正文
- 访问属性

这说明外部使用者拿到的是“统一内容语义”，而不是三套碎片数据。

### 15.6 为什么这轮必须强调统一

因为如果不统一，系统后面会越来越碎。

你会不断遇到这种问题：

- 文件能分享，字符串不能分享
- HTML 能 public，Markdown 只能 owner 看
- owner 能编辑，share 不能复用
- 页面一多就开始各自解释正文

统一的价值就在这里：

**它把“新增输入形式”控制在内容对象框架内，而不是让系统多长出一套平行能力。**

所以“统一”不是口号，而是防止系统继续分叉的工程手段。

## 16. 最后压缩成两句话

### 16.1 Markdown 字符串完整链路

就是：

**写入 -> 认 owner -> 校验 -> 生成内容身份 -> 保留原文 -> 生成最终 HTML -> 入内容库 -> owner/public/share 复用 -> 再受访问控制约束**

### 16.2 什么叫统一

就是：

**虽然文件、HTML、Markdown 的输入形式不同、物理存储也不完全一样，但系统在内容身份、查询、展示、分享、访问控制这些关键层面，把它们当成同一种内容对象管理。**

## 17. Markdown 转 HTML 具体是怎么实现的

这里讲的不是“抽象上怎么实现”，而是结合当前项目真实代码，说明 Markdown 转 HTML 这一段到底是怎样跑起来的。

先说结论：

- 核心代码在 [service.js](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/src/content/service.js)
- 真正默认渲染器是 `defaultMarkdownRenderer(markdown)`
- 调度入口是 `buildRenderedContent({ body, bodyFormat, markdownRenderer })`
- 创建时在 `createHtmlContent(...)` 里触发
- 更新时在 `updateContent(...)` 里再次触发
- 最终 HTML 会落到 `html_content`
- 对外再统一映射成 `renderedBodyHtml` 和 `htmlContent`

### 17.1 最外层入口：什么时候会触发 Markdown 渲染

Markdown 转换不是每次页面访问都触发，而是在**内容创建**和**内容更新**时触发。

核心判断点是：

- 如果 `bodyFormat === 'html'`，就直接把正文当作最终 HTML
- 如果 `bodyFormat === 'markdown'`，就调用 Markdown 渲染器先转一次

这个调度逻辑在 `buildRenderedContent({ body, bodyFormat, markdownRenderer })` 里。

你可以把这个函数理解成：**正文格式分发器**。

它不负责逐行解析 Markdown，它负责决定：

- 这次正文到底要不要进入 Markdown 渲染器
- 最后应该返回什么形态的数据

逻辑大意就是：

- `html` 路径：不做 Markdown 转换
- `markdown` 路径：进入 `markdownRenderer(body)`

而默认 `markdownRenderer`，就是 `defaultMarkdownRenderer`。

### 17.2 Markdown 真正在哪个函数里被转

真正做转换的是：

- `defaultMarkdownRenderer(markdown)`

位置在 [service.js](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/src/content/service.js)。

这个函数就是当前项目的内建 Markdown 渲染器。

也就是说，我们不是把 Markdown 原文交给某个浏览器库，也不是交给数据库，而是在 Node 服务里，自己把字符串解析后拼成 HTML。

### 17.3 `defaultMarkdownRenderer` 的整体结构是什么

这个函数内部大致分成 3 层：

1. 预处理
2. 块级解析
3. 行内解析

### 17.4 预处理

一开始它先做的是：

- 检查 `markdown` 是否是字符串
- 统一换行符
- 去掉首尾空白
- 空字符串直接报错

逻辑大意是：

```js
if (typeof markdown !== 'string') {
  throw ...
}

const normalized = markdown.replace(/\r\n/g, '\n').trim();

if (!normalized) {
  throw ...
}
```

这一步的作用是：

- 防止传入不是字符串
- 防止空正文
- 避免 Windows 换行 `\r\n` 影响后面的逐行解析

所以这里是在做：**让输入变成一个适合被解析的标准文本。**

### 17.5 块级解析

块级解析可以理解成：先按“大段结构”拆 Markdown，而不是一上来就处理粗体、链接这些小符号。

因为一段 Markdown 里，最先要判断的通常是：

- 这是不是代码块
- 这是不是标题
- 这是不是引用
- 这是不是列表
- 这是不是表格
- 如果都不是，那它就是普通段落

当前实现就是沿着这个思路，一行一行往前扫，再按规则吃掉对应的区块。

#### 17.5.1 围栏代码块

像这种内容：

```md
```js
const x = 1;
```
```

系统会先识别开头的围栏标记，再持续收集后面的行，直到遇到结束围栏。

然后输出成类似：

```html
<pre><code class="language-js">const x = 1;</code></pre>
```

这里会顺手做两件事：

- 保留语言标记，方便后续样式识别
- 对代码内容做 HTML 转义，避免代码里的 `<script>` 之类被当成真实标签执行

#### 17.5.2 分隔线

如果一整行匹配 `---`、`***` 这类水平分隔线语法，就会直接转成：

```html
<hr>
```

#### 17.5.3 数学公式块

项目里对块级公式也做了识别。像：

```md
$$
E = mc^2
$$
```

会被包成专门的数学块 HTML 容器，而不是当普通段落处理。

这说明当前渲染器不是只支持最基础 Markdown，而是已经把公式这种结构也纳入了正文渲染能力。

#### 17.5.4 标题

如果一行以 `#`、`##`、`###` 开头，系统会识别标题层级，然后转成：

- `<h1>`
- `<h2>`
- `<h3>`

这类 HTML 标签。

这里不是简单替换一个字符，而是要先判断前缀里有几个 `#`，再取出后面的标题文本，并继续走行内解析。

也就是说，标题里的粗体、链接、行内代码，后面还会继续处理。

#### 17.5.5 引用块

如果一行以 `>` 开头，系统会把它识别成引用。

例如：

```md
> 这是引用
```

会转成类似：

```html
<blockquote><p>这是引用</p></blockquote>
```

如果有多行连续引用，渲染器会把它们作为同一个引用区块来组织，而不是拆成互不相关的小片段。

#### 17.5.6 列表

无序列表和有序列表都会在块级阶段被识别。

比如：

```md
- 第一项
- 第二项
```

或者：

```md
1. 第一步
2. 第二步
```

系统会分别输出：

- `<ul><li>...</li></ul>`
- `<ol><li>...</li></ol>`

如果存在嵌套列表，渲染器也会根据缩进关系尽量组织正确层级，这也是为什么 owner/public/share 页面能展示你示例里那种多层列表，而不是把它们挤成一坨普通文本。

#### 17.5.7 表格

当前实现还支持 Markdown 表格。

像：

```md
| 列1 | 列2 |
| --- | --- |
| A   | B   |
```

会被识别成表头和表体，然后转成标准表格 HTML，例如：

```html
<table>
  <thead>...</thead>
  <tbody>...</tbody>
</table>
```

这也是为什么项目里的 Markdown 示例不仅有标题和列表，还有表格，也能在页面上被正常排版出来。

#### 17.5.8 普通段落

如果一段文本不属于上面任何一种块结构，它最后就会被当成普通段落。

也就是包成：

```html
<p>...</p>
```

但这里的 `...` 还不是原样塞进去，它还要继续经过下一层的行内解析。

### 17.6 行内解析

块级解析完成后，系统还要处理每个块内部的小语法。

这一层主要解决的是：

- 链接
- 图片
- 行内代码
- 粗体
- 斜体
- 行内公式
- 裸 URL

你可以把它理解成：

- 块级解析决定“这一整段是什么结构”
- 行内解析决定“这一段里面的局部语义是什么”

### 17.7 为什么要先 `escapeHtml`

当前实现里有一个非常关键的基础动作：先把原始文本做 HTML 转义。

意思是：如果用户正文里本来就有 `<`、`>`、`&` 这些字符，先不要让它们直接变成可执行 HTML，而是先转成安全文本。

例如：

- `<` 变成 `&lt;`
- `>` 变成 `&gt;`

这样做的目的，是避免用户原始输入里的危险标签直接混进最终页面。

这一点非常重要，因为 Markdown 渲染不是只求“显示漂亮”，还要保证“展示安全”。

### 17.8 为什么还需要 `stash(html)`

当前实现里还有一个很值得理解的小技巧，就是把已经生成好的 HTML 片段暂存起来。

你可以把 `stash(html)` 理解成：

1. 先给这段已经确定安全的 HTML 发一个占位符编号
2. 后续字符串替换先绕开它
3. 等所有替换都做完，再把原来的 HTML 片段放回去

为什么要这样？

因为行内解析往往要做很多轮正则替换：

- 先替换图片
- 再替换链接
- 再替换代码
- 再替换粗体斜体

如果没有暂存机制，前面刚生成出来的 `<a>`、`<img>`、`<code>`，后面可能又被下一轮替换误伤，最后把 HTML 结构搞乱。

所以 `stash(html)` 本质上是在解决一个工程问题：

**多轮文本替换时，如何保护已经生成好的 HTML 片段不被后续规则再次污染。**

### 17.9 行内都处理了哪些语法

按当前项目实现，大致包括下面这些。

#### 17.9.1 图片

Markdown 图片语法：

```md
![alt](url)
```

会转成 `<img ...>`。

这一步通常会把：

- 图片地址放进 `src`
- 说明文本放进 `alt`

这样页面里就能直接显示图片，而不是只剩一串原始 Markdown 符号。

#### 17.9.2 链接

像：

```md
[OpenAI](https://openai.com)
```

会转成：

```html
<a href="https://openai.com">OpenAI</a>
```

这里通常还会顺手加一些安全属性，避免外链带来不必要的页面控制风险。

#### 17.9.3 行内公式

像：

```md
$a^2 + b^2 = c^2$
```

会被识别成行内数学表达式，并包成对应的 HTML 结构，而不是当普通美元符号文本处理。

#### 17.9.4 裸 URL

如果正文里直接出现：

```text
https://example.com
```

哪怕作者没有手写 Markdown 链接语法，系统也会把它识别成可点击链接。

这会提升展示体验，因为很多人在写内容时确实只是把网址直接贴进去。

#### 17.9.5 行内代码

像：

```md
使用 `npm test` 执行测试
```

会把反引号里的部分包成 `<code>`。

这样页面展示时，命令、变量名、字段名都会更清晰。

#### 17.9.6 粗体和斜体

像：

- `**粗体**`
- `*斜体*`

会分别转成类似：

- `<strong>...</strong>`
- `<em>...</em>`

所以最终页面里会保留 Markdown 原来的强调语义，而不是只剩普通文字。

### 17.10 转换结果最后怎么进入存储

Markdown 被转成 HTML 后，并不是只在函数里临时返回一下就结束了。

后面还会进入一个存储映射过程。

当前项目里对应的是：

- `buildRichTextStorageFields({ body, bodyFormat, renderedBodyHtml })`

你可以把它理解成：

**把上层内容语义，翻译成底层数据库字段。**

例如它会整理出：

- 原始正文存哪里
- 正文格式存哪里
- 最终 HTML 存哪里

最后形成这类底层字段：

- `body_source`
- `body_format`
- `html_content`

这就完成了从“业务层正文对象”到“持久化字段”的落地。

### 17.11 创建时怎么接上这一步

在创建流程里，对应函数是：

- `createHtmlContent(...)`

它会把用户传进来的正文参数拿过来，先做格式判断，再调用：

- `buildRenderedContent(...)`
- `buildRichTextStorageFields(...)`

最后再把结果写进内容记录。

所以创建链路不是：

- 收到 Markdown -> 直接存库

而是：

- 收到 Markdown -> 渲染 HTML -> 组装存储字段 -> 再存库

### 17.12 更新时怎么接上这一步

更新流程对应的是：

- `updateContent(...)`

只要 owner 修改了：

- `body`
- `bodyFormat`

系统就会重新跑一次正文加工。

也就是说，更新不是只改一个字段值，而是重新计算这条内容的“最终展示结果”。

这就是为什么 owner 后续把 Markdown 正文改掉以后，详情页、公网页、分享页也会一起看到新的渲染结果。

### 17.13 查询时为什么能同时看到 `body` 和 `renderedBodyHtml`

到了查询阶段，系统会再做一次“对外语义映射”。

底层存储字段可能叫：

- `body_source`
- `body_format`
- `html_content`

但对外返回时，系统会整理成更稳定、更好懂的字段：

- `body`
- `bodyFormat`
- `renderedBodyHtml`
- `htmlContent`

这样前端和脚本就能同时拿到：

- 原始正文
- 正文格式
- 最终 HTML

所以 owner 详情页才能一边保留可编辑原稿，一边展示最终排版结果。

### 17.14 为什么说这套实现是“服务端预渲染”，不是“前端临时转换”

因为真正的 Markdown 转 HTML 发生在 Node 服务里，而且发生在写入和更新阶段。

页面层做的事更像是：

- 从服务端拿已经准备好的 HTML
- 再把它展示出来

这和很多前端现场临时解析 Markdown 的做法不同。

当前项目的方案有几个明显好处：

- owner/public/share 更容易保持一致
- 页面层更轻
- 验收脚本可以直接校验服务端结果
- 调试时更容易定位是“写入问题”还是“展示问题”

### 17.15 用一句话概括这套实现

当前项目里，Markdown 转 HTML 的本质实现是：

**在内容服务层，由 `buildRenderedContent` 按 `bodyFormat` 调度 `defaultMarkdownRenderer`，把原始 Markdown 解析成安全的结构化 HTML，再通过存储映射写入 `html_content`，最后在查询和页面层统一以 `renderedBodyHtml/htmlContent` 的形式复用。**

## 18. 当前密码访问是怎么实现的

这一节讲的不是“需求上支持密码访问”，而是当前项目里，这件事在系统内部究竟怎么落地。

先说结论：

- 密码不是挂在某个页面上的临时输入框
- 密码能力是挂在 `contents` 记录上的内容级访问规则
- public 入口和 share 入口共用同一套密码校验原则
- 校验通过后，会给浏览器一个短期访问态
- 校验失败后，不是直接把用户踢死，而是回到密码页继续重试

### 18.1 密码能力先落在 `contents` 字段上

当前项目里，密码访问的底座不是前端状态，而是 `contents` 里的三个字段：

- `access_mode`
- `access_password_hash`
- `access_hint`

它们的职责分别是：

- `access_mode`
  说明这条内容现在到底是 `public` 还是 `password`

- `access_password_hash`
  保存密码的 hash，而不是明文

- `access_hint`
  保存给访客看的提示语

这一步很重要，因为它意味着：

**密码保护是内容对象自己的属性，不是某个页面额外拼出来的能力。**

### 18.2 owner 在创建和更新时怎么设置密码

owner 创建或更新内容时，可以同时提交：

- `accessMode`
- `accessPassword`
- `accessHint`

服务层会先把这些输入标准化，然后决定怎么写入 `contents`。

对应逻辑在 [service.js](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/src/content/service.js) 的创建和更新流程里。

大致规则是：

- 如果 `accessMode=public`
  说明这条内容公开访问

- 如果 `accessMode=password`
  说明这条内容必须带访问密码

当模式是 `password` 时，系统不会保存明文密码，而是先对密码做 hash，再写入：

- `access_password_hash`

同时把提示语写到：

- `access_hint`

如果 owner 后续把内容从 `password` 切回 `public`，系统会把：

- `access_password_hash` 清空
- `access_hint` 清空

这说明密码保护不是永远残留的脏状态，而是受当前访问模式控制的正式字段。

### 18.3 为什么数据库里存的是 hash，不是明文密码

这是一个很基础但非常关键的安全点。

如果系统把密码明文直接存到数据库里，会带来几个问题：

- 管理后台的人一眼就能看到原密码
- 日志或调试信息泄漏时风险更高
- 后续数据导出、备份时也会把明文带出去

所以当前实现里，密码进入服务层后不会原样保存，而是通过 `hashPassword(...)` 转成 hash 形式。

后续校验时，也不是“拿用户输入的密码直接和数据库字符串做相等比较”，而是走：

- 用户输入密码
- 系统按相同规则处理
- 用 `verifyPassword(...)` 比对它和已存 hash 是否匹配

这才是正常的密码校验方式。

### 18.4 public 入口和 share 入口为什么要共用一套规则

这一点是这轮实现里很关键的地方。

很多系统容易做成：

- public 页面自己判断一套密码
- share 页面再判断另一套密码

这样很容易出现一边修了、一边漏了，最后形成绕过漏洞。

当前项目的做法是先站在“内容对象”这一层判断：

- 这条内容是不是 `password` 模式

如果是，那么无论你是从：

- `/web/public/content/:contentHash`
- `/api/public/content/:contentHash`
- `/web/public/share/:shareHash`
- `/api/public/share/:shareHash`

这些入口里的哪一种进来，最终都得接受同一原则：

- 没验证过，就先去密码页/密码校验
- 验证通过后，才允许继续拿正文

也就是说，入口可以不同，但访问控制规则必须统一。

### 18.5 系统怎么判断一条内容到底需不需要密码

当前服务层不是只看 `access_mode` 这个字段字面值，还会结合密码 hash 是否真实存在一起判断。

你可以理解成：

- 只有当 `access_mode=password`
- 并且 `access_password_hash` 里确实有有效值

系统才把这条内容当成真正的密码保护内容。

如果模式写成 `password`，但 hash 是空的，那从保护强度上讲其实是不成立的。

服务层当前会把这种情况收敛回一个更安全、更稳定的“有效访问模式判断”。

这类逻辑的意义是：

**系统最终信的不是表面配置，而是“这条保护链路是否真的闭合”。**

### 18.6 访客输入密码时，系统在校验什么

当访客来到密码保护内容页面时，页面不会自己偷偷决定是否放行。

真正的校验动作发生在服务端：

1. 先找到这条内容
2. 看它是否处于密码模式
3. 读取 `access_password_hash`
4. 取出用户刚刚输入的密码
5. 用 `verifyPassword(...)` 判断是否匹配

如果不匹配，就拒绝这次访问。

如果匹配，才认为这次密码验证成功。

所以这里真正被校验的是：

- “用户输入”
- 和 “这条内容保存的密码 hash”

是否构成同一组密码。

### 18.7 校验通过后，为什么还要发短期访问态

如果系统每次访问 public/share 页面都要求重新输密码，体验会很差。

比如用户刚输完密码看正文，点一下刷新，又被要求再输一次，这在实际使用里会非常别扭。

所以当前实现不是“这一刻校验通过就完了”，而是：

- 校验通过后
- 给当前浏览器一个短期有效的访问状态

你可以把它理解成：

**密码不是只开一次门，而是换来一张短时间内可重复通行的小票。**

这张“小票”不是 owner 登录态，也不是永久授权，而是：

- 只针对某条内容
- 只在短时间内有效
- 只让访客继续访问这一条受保护内容

### 18.8 这个短期访问态在实现上是什么思路

从实现思想上看，它通常会绑定几个维度：

- 当前浏览器
- 当前内容
- 当前有效期

所以它不是“全站通行证”，而是“某个浏览器对某条内容的短期访问许可”。

项目当前 public/share 的密码闭环，就是依赖这层访问态来避免重复输密码。

你可以把它理解成：

- owner 登录态解决“你是谁”
- 密码访问态解决“你刚刚通过了哪条公开内容的密码门禁”

这两者不是一回事。

### 18.9 错误密码为什么还能重试

这是用户体验上看起来很小、但实现上必须想清楚的一点。

错误密码时，系统没有把用户直接打到一个死的 403 页面，而是把它设计成：

- 仍停留在密码页
- 页面下方显示错误提示
- 用户可以立即再次输入密码重试

这背后其实是在区分三种状态：

1. 还没验证过
2. 验证失败
3. 验证成功

如果系统不区分这三种状态，就很容易出现：

- 错一次就直接终止
- 用户只能后退或重开页面
- 前端根本不知道该怎么继续

现在的实现要求里，“可重试”本质上就是：

**验证失败不是终止态，而是一种允许继续恢复的中间态。**

### 18.10 红字错误提示是怎么接上来的

当密码错误时，服务端不会把这次访问当作成功，也不会发访问态。

然后页面层会重新渲染密码输入页，并带上一条明确错误信息。

这样浏览器里看到的效果才会是：

- 你还在原来的密码框页面
- 密码框下面有红字提示“密码错误”或同类错误文案
- 你可以继续输入下一次

所以这个红字提示不是孤立的 UI 装饰，它依赖的是：

- 服务端知道这次验证失败
- 页面层能把失败状态重新渲染回来

### 18.11 public 和 share 在密码能力上的差别到底是什么

它们的差别主要不在“密码怎么验”，而在“先怎么找到内容”。

#### public 内容入口

public 内容入口通常是：

- 先按 `content_hash` 找到 `contents`
- 再判断这条内容是不是密码模式

#### share 入口

share 入口通常是：

- 先按 `share_hash` 找到 `share_links`
- 再根据它指向的 `content_id` 找到 `contents`
- 然后再判断这条内容是不是密码模式

也就是说：

- 找内容的路径不同
- 但真正的密码门禁判断仍然落在内容对象上

### 18.12 当前密码能力和正文链路是怎么汇合的

这轮实现里，密码能力不是独立子系统，而是附着在同一条内容主链路上的。

一条 Markdown 或 HTML 内容在创建完成后，已经有：

- 原始正文
- 最终展示 HTML
- content hash
- owner 身份

然后再叠加：

- `access_mode`
- `access_password_hash`
- `access_hint`

这样这条内容就从“能展示”，升级成“能按 public/password 两种模式被展示”。

所以密码功能不是和正文无关的一层壳，而是内容对象生命周期的一部分。

### 18.13 用一句话概括当前密码实现

当前项目里的密码访问，本质上是：

**在 `contents` 记录上保存访问模式、密码 hash 和提示语，在 public/share 两类入口统一按内容级规则做服务端校验；校验通过后发短期访问态，校验失败则回到密码页并允许带错误提示继续重试。**
