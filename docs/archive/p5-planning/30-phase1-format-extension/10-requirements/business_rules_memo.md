# 业务规则备忘录

## 1. 需求概述

当前系统已完成 P5 第一阶段内容对象接入，具备统一 `content` 语义写入、查询、公开详情展示和 owner 列表增强能力。

下一阶段的目标是在不重构现有内容主链路的前提下，为正文增加格式声明能力，并新增 Markdown 输入支持，使调用方既可以继续提交 HTML，也可以直接提交 Markdown，系统负责将其转换为最终可展示 HTML，并继续沿用现有展示骨架。

## 2. 核心目标

1. 为内容对象正文增加明确的格式声明。
2. 支持 `html` 与 `markdown` 两种正文格式输入。
3. 对 `markdown` 输入提供真实转换能力，而不是只做格式标记。
4. 同时保留原始正文、正文格式和最终展示 HTML。
5. 继续复用现有 `rich_text` 展示链路，不引入第二套公开访问体系。

## 3. 范围内内容

1. 为内容对象新增正文格式声明字段，例如 `bodyFormat`。
2. 支持 `html` 正文原样进入现有展示链路。
3. 支持 `markdown` 正文转换为最终展示 HTML。
4. 写入链路中同时保留原始正文与最终展示 HTML。
5. 查询链路中返回原始正文、格式声明与最终展示 HTML。
6. owner 页面与 public 页面继续消费最终展示 HTML。
7. 支持 HTML 或 Markdown 中引用已有可访问资源，例如外部图片 URL。

## 4. 范围外内容

1. 图片上传、图片托管、图片清理生命周期。
2. 附件型资源模型扩展。
3. 第二套平行内容存储模型。
4. 第二套 `rich_text` 公开访问体系。
5. 全局接口或字段命名重构。
6. 立即废弃现有 `POST /api/write/html`。
7. 复杂富文档系统、块编辑器、结构化内容模型。

## 5. 核心概念定义

### 正文格式声明

指内容对象用于说明正文原始输入格式的字段。当前阶段仅考虑两种取值：`html` 与 `markdown`。

### 原始正文

指调用方提交的原始正文内容。若 `bodyFormat=markdown`，则原始正文为 Markdown 文本；若 `bodyFormat=html`，则原始正文为 HTML 文本。

### 最终展示 HTML

指系统最终用于 owner 页面、public 页面和其他展示型链路的可直接渲染 HTML 内容。

### 已有可访问资源引用

指正文中通过 `<img src="...">` 或 `![alt](url)` 等方式引用的已有 URL 资源。系统仅负责渲染引用，不负责上传、托管和清理这些资源。

## 6. 用户与使用场景

当前主要用户仍为后台操作者 / 内容发布者。

典型场景包括：

1. 调用方直接提交 HTML 正文并展示。
2. 调用方直接提交 Markdown 正文，由系统转换后展示。
3. 内容页面与公开详情页继续展示最终渲染后的 HTML。
4. 后续编辑或调试场景中，需要保留原始正文与格式声明用于回看或再处理。

## 7. 功能性要求

1. 写入内容对象时必须能够声明正文格式。
2. 系统必须支持 `html` 与 `markdown` 两种正文格式。
3. 当输入为 `markdown` 时，系统必须将其转换为最终展示 HTML。
4. 系统必须同时保留原始正文和最终展示 HTML。
5. 查询结果必须返回：
   - 原始正文 `body`
   - 正文格式 `bodyFormat`
   - 最终展示 HTML，例如 `renderedBodyHtml`
6. owner 列表、owner 详情、public 详情必须继续使用最终展示 HTML 或其派生结果。
7. 现有 `POST /api/write/html` 必须继续可用。
8. 新能力主路径应继续优先通过统一 `content` 语义承载。

## 8. 非功能性要求

1. 必须保持与现有 `contents` 模型和 `rich_text` 展示骨架兼容。
2. 不得因 Markdown 支持而破坏现有 HTML 内容的展示结果。
3. 不得为了支持 Markdown 引入第二套公开访问体系。
4. 不得在本阶段引入资源托管和复杂附件生命周期。
5. 新增转换能力应保持输出可直接供现有页面消费。

## 9. 业务规则

1. 正文格式当前仅允许 `html` 与 `markdown`。
2. `html` 输入继续按现有语义原样展示。
3. `markdown` 输入必须转换为最终展示 HTML 后再进入展示链路。
4. 系统必须同时保留：原始正文、正文格式、最终展示 HTML。
5. 查询结果必须包含原始正文、正文格式和最终展示 HTML，避免展示链路与查询链路再次分叉。
6. 页面与公开详情默认消费最终展示 HTML，而不是直接消费 Markdown 原文。
7. 图片与外部资源当前只支持引用已有可访问 URL，不进入托管与清理责任。
8. 现有 `write/html` 作为兼容接口继续保留，不要求下一阶段立即下线。

## 10. 边界条件

1. 当前阶段不支持除 `html` / `markdown` 之外的正文格式。
2. 当前阶段不支持图片上传与附件托管。
3. 当前阶段不支持把 Markdown 之外的通用文档格式自动转换为 HTML。
4. 当前阶段不支持平台级内容模型重构。
5. 当前阶段允许 HTML 或 Markdown 中引用外部可访问图片，但不保证所有外部资源始终可用。

## 11. 异常处理

1. 当正文格式缺失或不在允许范围内时，系统必须拒绝写入。
2. 当 Markdown 转换失败时，系统必须返回明确失败，而不是写入不确定结果。
3. 当引用的外部图片或资源不可访问时，系统当前不做业务拦截，页面按浏览器默认失败表现处理。
4. 当内容不存在时，查询与公开访问应继续沿用现有“不存在 / 不可访问”处理规则。

## 12. 数据约束

1. 每条内容必须有原始正文 `body`。
2. 每条内容必须有正文格式 `bodyFormat`。
3. 每条内容必须有最终展示 HTML，例如 `renderedBodyHtml`。
4. 当 `bodyFormat=markdown` 时，`body` 保存原始 Markdown，`renderedBodyHtml` 保存转换后的 HTML。
5. 当 `bodyFormat=html` 时，`body` 保存原始 HTML，`renderedBodyHtml` 可等于原始 HTML 或其等价展示结果。
6. 现阶段不要求额外建立图片资源表或附件关联表。

## 13. 待确认事项

1. `bodyFormat`、`renderedBodyHtml` 与现有底层字段的精确映射方案尚未固化。
2. Markdown 转换是否需要特定安全策略或清洗规则尚未固化。
3. 查询接口中是否保留现有 `htmlContent` 兼容字段，以及与 `renderedBodyHtml` 的关系尚未固化。
4. 现有 owner 更新接口是否同步支持 Markdown 输入尚待后续阶段确认。
