# P2-T1 MVP Scope Freeze

## 1. 目的

本文档用于冻结 `P2-T1` 的范围口径，解决两个后续实现期会反复摇摆的问题：

- 哪些内容仍然属于“真正 MVP”的收尾缺口。
- 原始需求中的“外部用户”在当前项目中应如何解释。

本文件一旦确认，后续 `P2-T3` 与 `P2-T4` 必须以此为准推进，不再回到“是否要做”的模糊状态。

## 2. 冻结结论

### 2.1 对“外部用户”的定义

冻结结论：当前项目中的“外部用户”按 `公开访客` 解释，而不是 owner 侧用户。

理由：

- 原始需求中 owner 与外部访问者是分开的角色。
- `business_rules_memo` 中已经把“外部访问者”定义为通过 HTTP 访问内容的只读用户。
- 如果把“外部用户”解释成 owner 侧用户，则需求 3/4/5 会与现有 owner 查询能力混淆，无法体现“对外发布服务”的项目目标。

因此：

- 需求 3 的列表页，指公开访客可以访问的公开内容列表。
- 需求 4 的搜索页，指公开访客可以访问的公开内容搜索。
- 需求 5 的详情页，指公开访客从公开发现流进入的内容详情，不只是拿到 hash 后的直达页。

### 2.2 仍属于 MVP 收尾的内容

冻结结论：以下内容继续视为 MVP 收尾，不下放到 Phase 3。

1. 文件原样下载一致性验证。
2. 至少一种真实文件类型的下载后可打开 / 可读取验证。
3. 面向公开访客的公开内容列表页。
4. 面向公开访客的公开内容搜索页。
5. 从公开列表 / 公开搜索进入公开详情页的闭环。

### 2.3 不属于当前 MVP 收尾的内容

冻结结论：以下内容不属于当前 MVP 收尾，进入后续阶段处理。

- 完整 owner 产品化前端。
- 更自然的 owner 登录态与凭据管理。
- 内容编辑、批量操作、搜索增强。
- 浏览器级全量 E2E 体系。
- 大规模 UI 重构。

## 3. 对原始 6 条 MVP 需求的冻结判定

### 3.1 需求 1

`允许我将一个文件保存到此应用中，并允许外部用户通过 http 访问此文件`

冻结判定：未完全闭合。

尚缺：

- 原样下载一致性验收
- 真实文件类型交付验证

### 3.2 需求 2

`允许我将一段富文本保存到此应用中，并允许外部用户通过 http 查看此文本`

冻结判定：已基本闭合。

说明：

- 当前 HTML 富文本写入与公开访问主链路已实测通过。

### 3.3 需求 3

`外部用户可以在页面的内容列表中找到我发布的内容`

冻结判定：未完成。

尚缺：

- 面向公开访客的公开列表页
- 仅公开内容可见的过滤规则

### 3.4 需求 4

`外部用户可以在页面中搜索我发布的内容`

冻结判定：未完成。

尚缺：

- 面向公开访客的公开搜索页
- 公开范围搜索边界

### 3.5 需求 5

`外部用户可以查看我发布的内容的详情`

冻结判定：部分完成，但未完全闭合。

已具备：

- 直达式 public 详情页

尚缺：

- 从公开列表 / 搜索进入详情的完整公开流

### 3.6 需求 6

`我可以通过 open api 向应用中保存文件或是富文本`

冻结判定：已闭合。

## 4. 对后续任务的直接约束

### 4.1 对 `P2-T3` 的约束

`P2-T3` 不是“补一个可选增强验证”，而是补足需求 1 是否真正成立的正式验收项。

最低要求：

- 至少一组字节级一致性验证。
- 至少一组真实文件类型的上传后下载回读验证。

### 4.2 对 `P2-T4` 的约束

`P2-T4` 不再允许输出“保留讨论”类结果。

最低要求：

- 必须实现公开列表 / 公开搜索 / 公开详情闭环。
- 必须明确只暴露已公开内容。
- 必须把 owner 侧页面与公开访客页面严格分开。

### 4.3 对 Phase 3 的约束

Phase 3 不得再承接本文件列出的 MVP 收尾缺口，否则会导致 MVP 与产品化范围重新混淆。

## 5. 当前项目阶段结论

冻结结论如下：

- 当前系统已完成“技术主链路型 MVP”的大部分能力。
- 但按原始需求定义，真正 MVP 仍未完全完成。
- `P2-T1` 之后，项目应先补齐 `需求 1 / 3 / 4 / 5` 尚未闭合的部分，再进入纯稳定化意义上的 Phase 2 和产品化意义上的 Phase 3。
