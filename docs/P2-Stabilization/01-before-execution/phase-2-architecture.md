# Phase 2 Architecture

## 1. 本轮架构目标

Phase 2 的目标不是继续做演示型能力，而是把当前 PocketBase + Node 业务壳推进到“可稳定运行、可重复部署、可明确诊断问题”的状态。

本轮架构需要解决三类问题：

- 仍属于 MVP 的缺口如何收口，尤其是文件原样交付与公开访客发现流是否补齐。
- 当前运行链路如何从“手工启动 + 手工核对”提升为“可自检、可诊断、可冷启动复现”。
- 现有业务壳、PocketBase 集合、文件落盘之间的依赖如何稳定化，避免后续 Phase 3 产品化时反复返工。

## 2. 本轮明确不采用的方向

- 不在 Phase 2 内重做底座，继续沿用 `PocketBase + 自研 Node 服务`。
- 不在 Phase 2 内建设完整 owner 产品化前端。
- 不在 Phase 2 内引入复杂消息队列、对象存储或多节点部署架构。
- 不在 Phase 2 内默认引入浏览器级全量 E2E 体系，除非为关键公开链路补一小组高价值用例。

## 3. 结构边界

Phase 2 建议维持并增强以下 5 个边界：

### 3.1 Runtime Bootstrap Boundary

负责本地环境启动前检查，包括：

- 环境变量完整性检查
- 工作目录可写性检查
- PocketBase 二进制与数据目录状态检查
- 端口占用与服务存活检查
- 必要集合/字段状态检查

该边界只负责“启动前判断”和“问题显式报出”，不直接承接业务流。

### 3.2 Content Delivery Boundary

负责文件与富文本对外 HTTP 交付的一致性定义，包括：

- 文件下载响应头与文件名策略
- 文件落盘内容与回传内容一致性验证
- HTML 富文本渲染与沙箱策略
- 公开链接、公开详情、公开访问控制

这是 Phase 2 中最直接承接“MVP 收尾缺口”的边界。

### 3.3 Public Discovery Boundary

负责“外部用户是否能够发现已发布内容”的公开流边界，包括：

- 公开列表
- 公开搜索
- 从公开列表/搜索进入详情
- 公开范围过滤，只暴露已发布/可访问内容

如果项目确认原始需求中的“外部用户”就是公开访客，则此边界属于 MVP 缺口补完的一部分；若后续另有更严格定义，也至少要在 Phase 2 中把边界显式化。

### 3.4 Operational Observability Boundary

负责运行期可诊断性，包括：

- 启动日志分层
- PocketBase 请求失败上下文
- 文件写入/删除失败上下文
- 公开访问失败分类
- 人工排障手册所依赖的最小观察信号

该边界不做复杂监控平台接入，只做本地部署场景足够使用的诊断能力。

### 3.5 Verification Contract Boundary

负责把“手工验通过”和“真正可持续回归验证”区分开，包括：

- 冷启动检查清单
- 文件原样下载一致性检查
- MVP 公开链路回归脚本
- 文档化的人工验收路径

此边界的重点是让后续每次改动都知道最少要验证什么。

## 4. 模块职责

### 4.1 `scripts/` 启动与自检模块

负责：

- 统一服务启动入口
- 启动前快速失败
- 服务冲突提示
- 面向人工联调的明确命令输出

不负责：

- 业务数据修复
- 隐式修改 PocketBase 集合定义

### 4.2 `src/config.js` 与配置装载模块

负责：

- 环境变量读取
- 路径解析
- 基础格式校验
- 给上层暴露一致配置对象

### 4.3 `src/pocketbase/*` 或等价数据访问模块

负责：

- PocketBase admin 鉴权
- 集合查询与写入
- 集合状态检查所需最小读能力
- 统一 PocketBase 错误封装

### 4.4 `src/routes/public/*` 与公开访问模块

负责：

- 公开内容详情
- 公开文件下载
- 公开列表与公开搜索（若纳入本轮）
- 对只公开内容的过滤

### 4.5 `docs/P1-MVP` 与 `docs/P2-Stabilization` 文档模块

负责：

- 保存人工验收步骤
- 保存真实环境已知问题
- 区分 MVP 已验内容与下一阶段待补内容

## 5. 依赖方向

明确允许的依赖方向：

- 启动脚本 -> 配置装载模块
- 启动脚本 -> 基础服务探针 / 集合探针
- 路由层 -> 业务服务层 -> PocketBase 访问层
- 路由层 -> 文件存储层
- 验证脚本 -> HTTP API / public 页面

明确禁止的依赖方向：

- PocketBase 访问层反向依赖 HTTP 路由层
- 文档定义依赖实现中的隐式默认值而不显式说明
- 前端页面直接绕过业务服务层去拼装 PocketBase 内部规则
- 启动脚本在未告知用户的情况下悄悄修改生产数据

## 6. 与现有骨架的衔接方式

Phase 2 继续沿用当前 MVP 的骨架：

- PocketBase 作为底座，继续承接用户、内容、分享链接数据
- Node 业务壳继续承接 API Key 鉴权、内容统一建模、公开访问规则
- `scripts/` 继续作为本地启动与校验入口
- `docs/P1-MVP/02-after-execution/` 继续保存 MVP 实测材料

本轮不是换骨架，而是在现有骨架上补“自检、交付一致性、公开发现、可诊断性”。

## 7. 风险点与控制策略

### 7.1 PocketBase 集合实际状态漂移

风险：

- 真实环境中管理员手工改字段，导致脚本、代码、文档与实际集合状态不一致。

控制策略：

- 把关键字段约束写入显式检查项。
- 启动时输出可读的集合状态异常提示。

### 7.2 文件下载“看起来可用”但与原文件不一致

风险：

- 只验证了可访问，没有验证字节级一致性或真实文件打开能力。

控制策略：

- 把下载一致性验证提升为正式回归项。
- 选取至少一种真实文件类型做人工与脚本双重检查。

### 7.3 公开列表/搜索边界失控

风险：

- 若公开列表或搜索直接暴露非公开内容，会引发权限穿透。

控制策略：

- 明确只允许公开内容进入公开发现流。
- 单独为公开过滤写回归测试。

### 7.4 运维脚本过度自动修复

风险：

- 为了“方便启动”而隐式修改数据，导致线上状态不可预期。

控制策略：

- Phase 2 的脚本只做检查、提示、显式启动，不做静默修复。
