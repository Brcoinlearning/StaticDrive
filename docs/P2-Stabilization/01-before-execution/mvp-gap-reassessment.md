# MVP Gap Reassessment

本文档用于按原始需求重新衡量当前 MVP 的完成度，并把“仍属于 MVP 但尚未完全闭合”的缺口单独识别出来。

## 1. 重新定义后的 MVP

当前项目的真正 MVP 以原始需求为准，定义为以下 6 条：

1. 允许用户将一个文件保存到此应用中，并允许外部用户通过 HTTP 访问此文件。
2. 允许用户将一段富文本保存到此应用中，并允许外部用户通过 HTTP 查看此文本。
3. 外部用户可以在页面的内容列表中找到用户发布的内容。
4. 外部用户可以在页面中搜索用户发布的内容。
5. 外部用户可以查看用户发布内容的详情。
6. 用户可以通过 Open API 向应用中保存文件或富文本。

## 2. 当前完成度评估

### 2.1 已基本完成

#### 需求 2：富文本保存与 HTTP 查看

当前状态：已完成。

已具备：

- Open API 保存 HTML 富文本
- public 访问 HTML 内容
- 网页层安全渲染 HTML 内容

#### 需求 6：通过 Open API 保存文件或富文本

当前状态：已完成。

已具备：

- `POST /api/write/html`
- `POST /api/write/file`

### 2.2 已部分完成，但仍需补足为真正 MVP

#### 需求 1：文件保存并允许外部用户通过 HTTP 访问

当前状态：主链路已打通，但仍缺关键验证。

已具备：

- 文件可写入本地工作目录
- 可创建分享链接
- public 文件页可访问
- 可生成下载入口

尚未充分闭合的部分：

- 尚未把“下载回去的文件与原始文件完全一致”作为正式验收项固化
- 尚未验证典型真实富文本文件类型在下载后是否可正常打开/读取
- 当前文件访问方式仍偏 MVP 演示型，需要进一步明确是否满足最终 MVP 对“原样下载”的要求

#### 需求 3：外部用户可在页面内容列表中找到已发布内容

当前状态：若这里的“外部用户”指公开访客，则未完全完成。

当前系统已具备：

- owner 侧列表页
- owner 侧查询接口

当前系统未明确具备：

- 公开访客直接使用的公开列表页
- 面向公开访客的公开内容发现机制

#### 需求 4：外部用户可搜索已发布内容

当前状态：若这里的“外部用户”指公开访客，则未完全完成。

当前系统已具备：

- owner 侧搜索页
- owner 侧搜索接口

当前系统未明确具备：

- 公开访客直接使用的公开搜索页
- 公开范围内内容的搜索边界定义

#### 需求 5：外部用户可查看已发布内容详情

当前状态：部分完成。

当前系统已具备：

- 拿到 `content_hash` 或 `share_hash` 后可访问具体内容
- public HTML 页
- public 文件页

当前系统未明确具备：

- 从公开列表/公开搜索结果进入详情的完整公开访问流

## 3. 当前仍属于 MVP 的缺口

基于上述评估，以下内容不应简单归类为后续增强，而应视为“当前 MVP 尚未完全闭合”的缺口：

1. 文件原样下载一致性验证
2. 面向真实文件类型的 HTTP 文件交付验证
3. 若“外部用户”定义为公开访客，则需要公开内容列表页
4. 若“外部用户”定义为公开访客，则需要公开搜索页
5. 若“外部用户”定义为公开访客，则需要从公开列表/搜索进入详情的完整公开流

## 4. 当前结论

- 当前系统已经完成了“技术主链路型 MVP”的大部分内容。
- 但按原始需求重定义后的真正 MVP，仍存在文件交付一致性与公开内容发现能力方面的缺口。
- 后续规划不能直接假设“MVP 完整结束”，而应先决定这些缺口是否纳入本轮补完。
