# Phase 3 Tasks

## 1. 执行前提

Phase 3 只在以下条件满足后才成立：

- [p2-t1-mvp-scope-freeze.md](/Users/mr.hu/Desktop/开发项目/静态网页服务-文件管理/docs/P2-Stabilization/01-before-execution/p2-t1-mvp-scope-freeze.md) 已确认。
- 属于 MVP 收尾的文件交付缺口已经补齐。
- 属于 MVP 收尾的公开列表 / 搜索 / 详情闭环已经补齐。

Phase 3 不承接以下内容：

- 文件原样下载一致性验证
- 真实文件类型交付验证
- 面向公开访客的公开列表页
- 面向公开访客的公开搜索页
- 从公开列表 / 搜索进入公开详情页的闭环

## 3. 逐项说明

### P3-T1 冻结 owner 产品化范围

目标：

- 明确 Phase 3 只做哪些 owner 高频能力，避免前端范围无限膨胀。

输出结果：

- Phase 3 owner 能力清单

### P3-T2 owner 页面产品化

目标：

- 提供比当前 MVP 页面更完整、更可用的 owner 列表/搜索/详情体验。

输出结果：

- Owner 列表/搜索/详情页面
- 对应浏览器交互闭环

### P3-T3 内容更新与批量操作

目标：

- 支持已有内容更新、重新分享、批量删除或批量分享等能力。

风险标记：高。

原因：

- 写路径一旦扩展，容易带来权限与一致性回归。

### P3-T4 owner 凭据管理

目标：

- 让 owner 使用体验不再完全依赖手工维护 API Key。

风险标记：高。

原因：

- 这项会影响身份模型和后续安全边界。

### P3-T5 浏览器级关键 E2E

目标：

- 为 owner 与公开访客关键路径增加浏览器自动化回归。

输出结果：

- 一组高价值 E2E 用例
- 明确的执行命令

### P3-T6 文档收口

目标：

- 更新使用说明、前端验收说明与 Phase 3 完成记录。

## 4. 串行 / 并行关系

1. 先完成 P3-T1。
2. P3-T2、P3-T3、P3-T4 可在边界冻结后并行推进。
3. P3-T5 依赖前述产品能力基本成型。
4. P3-T6 最后收口。

## 5. 不允许进入实现期临时假设的说明

- 不允许默认 owner 前端等于“把现有接口换个页面壳”。
- 不允许在未定义身份模型前直接堆登录逻辑。
- 不允许用浏览器 E2E 替代现有 API / 集成验证。
- 不允许把 MVP 收尾缺口重新转移给 Phase 3。
