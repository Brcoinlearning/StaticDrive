# 任务拆解文档

## 0. 本周 MVP 执行说明

本周的主要目标不是按编号机械推进全部工程细节，而是优先打通“可演示、可部署到 MiniPC 前验证”的 MVP 主链路。

本周 MVP 目标链路：

1. 智能体或调用方携带 API Key 调用业务壳接口。
2. 成功上传文件或保存 HTML 富文本。
3. 数据写入 PocketBase，并保持统一 `Content` 模型。
4. 发布者可查看自己的列表、搜索结果和详情。
5. 外部用户可通过分享访问入口查看 HTML 或下载文件。
6. 网页层可完成基础展示。

基于该目标，本周执行顺序采用“MVP 冲刺顺序”，而不是严格按 task 编号串行推进：

- `T1`：先落地 PocketBase 底座与基础数据模型。
- `T2 + T3`：作为同一实现批次推进，尽快形成可启动业务壳和 API Key 身份上下文。
- `T4`：优先完成统一内容写入能力，确保文件与 HTML 进入统一 `Content` 模型。
- `T6`：优先补齐列表、搜索、详情查询，尽快验证内容主线是否成立。
- `T5`：在内容主线稳定后补齐分享链接与 hash 访问规则。
- `T7`：接入对外访问网页层，形成端到端演示链路。
- `T8`：在主链路跑通后补齐删除与撤销分享闭环。
- `T9`：最后统一完成集成验证、样例数据和部署说明。

执行约束：

- 不修改既有 task 定义、依赖和架构边界，只调整实现优先级。
- 若某项工作不能更快推动“写入 -> 可查询 -> 可访问 -> 可展示”的主链路，则不应抢占本周优先级。
- `T8` 和 `T9` 仍然是 MVP 完整交付的一部分，但优先级低于主链路打通。

## 2. task 详细定义

### T1 落地 PocketBase 底座并初始化基础数据模型

目标：

- 获取并启动 `PocketBase`，使其在当前项目环境中可运行。
- 确定底座运行方式，例如二进制运行、容器运行或等价落地方式。
- 初始化 `User`、`Content`、`ShareLink` 基础集合或等价数据结构。
- 建立字段、约束、索引和上传目录等基础存储配置。
- 明确 PocketBase 文件存储与本地工作目录的映射关系。
- 完成一次最小读写验证，确认后续业务壳可以稳定连接该底座。

输入依赖：

- `business_rules_memo`
- `tech_selection`

输出结果：

- 可运行的 PocketBase 基础环境。
- 数据模型定义、初始化脚本或等价初始化方案。
- 已确认的存储目录与工作目录映射方案。
- 一次最小化连通性验证结果。

说明：

- 该 task 不是“只安装一个开源项目”，而是把 `PocketBase` 作为本项目底座正式落地。
- 该 task 是所有后续实现的底座起点。

### T2 实现业务壳项目骨架与配置加载

目标：

- 创建独立于 PocketBase 的业务壳服务工程骨架。
- 接入环境配置、PocketBase 连接、本地工作目录配置。
- 建立基础 HTTP 路由、统一错误处理、日志和健康检查入口。
- 明确业务壳只承载业务规则与接口编排，不承载底层数据库替代实现。

输入依赖：

- T1

输出结果：

- 可启动的业务壳服务骨架。
- 一套明确的配置项约定，例如 PocketBase 地址、工作目录、监听端口、API Key 头部约定。
- 基础路由分组约定，至少区分写入接口、查询接口和前台访问接口。

说明：

- 该 task 的目标不是快速堆几个 handler，而是固定业务壳作为独立层的工程边界。
- 该 task 完成后，后续业务能力必须继续叠加在该骨架上，而不是绕回 PocketBase hook 或前台页面中实现。

### T3 实现 API Key 身份映射与权限中间层

目标：

- 实现 API Key 验证。
- 建立 API Key 到业务用户的映射机制。
- 为受保护接口加入统一身份上下文。

输入依赖：

- T2

输出结果：

- 可复用的认证中间层或服务模块。

### T4 实现统一内容写入能力（文件 + HTML）

目标：

- 提供文件上传 API。
- 提供 HTML 富文本保存 API。
- 两类写入最终落到统一 `Content` 模型，而不是形成两套数据主线。
- 生成 `content_hash`，写入 PocketBase，保持落盘与记录一致。
- 对文件类型明确处理文件名、存储路径、大小、类型等元数据。
- 对 HTML 类型明确处理标题、正文内容和可渲染字段。

输入依赖：

- T1
- T2
- T3

输出结果：

- 文件写入接口。
- 富文本写入接口。
- 统一内容写入服务。
- 统一内容写入结果结构，至少返回内容 ID、类型、hash 和访问地址。

风险标记：

- 高风险。该 task 同时触及文件系统、底层存储和统一模型边界。

说明：

- 不允许为文件和 HTML 分别建立风格完全不同的持久化流程，只允许在输入适配和最终渲染环节按类型分支。
- 若实现方案需要跳过统一 `Content` 模型，应判定为偏离架构，不得进入编码。

### T5 实现分享链接与 hash 路由解析

目标：

- 生成内容 hash 与分享 hash。
- 实现分享链接创建、撤销和解析逻辑。
- 校验外部访问合法性。
- 明确区分“内容标识 hash”和“分享访问 hash”的用途，不允许概念混用。
- 建立统一的 hash 解析入口，不允许由前台页面自行拼接或猜测底层路径。

输入依赖：

- T4

输出结果：

- 分享服务模块。
- hash 解析与访问校验模块。
- 清晰的路由约定，例如内容访问路由与分享访问路由的区分规则。

风险标记：

- 高风险。该 task 直接影响对外访问安全边界。

说明：

- 不允许把原始文件名、原始路径或 PocketBase 内部文件 URL 直接暴露给外部访问层。
- 不允许把 hash 解析逻辑散落在多个 handler 或页面组件中，必须集中在单一访问校验模块。

### T6 实现内容列表、搜索、详情查询接口

目标：

- 实现发布者可用的列表接口。
- 实现基础搜索接口。
- 实现详情查询接口。
- 查询范围必须默认限制在当前拥有者可见内容内，不允许默认返回跨用户结果。
- 排序、分页和搜索条件必须建立在统一 `Content` 模型之上。

输入依赖：

- T4

输出结果：

- 统一查询 API。
- 列表、搜索、详情三类查询的明确参数约定与返回结构。

说明：

- 可与 T5 并行推进，但不能早于 T4。
- 不允许为列表、搜索、详情分别走三套无关的数据访问实现。
- 不允许前台直接补查询逻辑来绕过业务壳接口缺口。

### T7 实现对外访问网页层

目标：

- 实现列表页、搜索页和详情页。
- 对 HTML 内容进行安全渲染。
- 对文件类型内容提供下载或预览入口。
- 页面只消费业务壳提供的接口，不自行实现权限、hash 解析或底层拼接规则。
- 页面结构必须围绕统一内容模型构建，而不是区分两套完全不同的信息架构。

输入依赖：

- T5
- T6

输出结果：

- 可访问的前台网页层。
- 与业务壳接口对齐的页面路由约定与页面状态处理方式。

说明：

- 网页层的目标是内容访问，不是后台管理替代品。
- 不允许在网页层中补写本应属于业务壳的权限判断、分享校验或数据拼装逻辑。

### T8 实现删除与撤销分享能力

目标：

- 实现内容删除。
- 实现分享链接撤销。
- 确保只有内容拥有者可执行操作。
- 删除内容时同时处理元数据、分享记录与物理文件之间的一致性。
- 撤销分享后，原分享访问入口必须立即失效。

输入依赖：

- T4
- T5

输出结果：

- 删除与撤销相关 API。
- 明确的删除后状态与撤销后访问行为约定。

说明：

- 不允许只删数据库记录而遗留物理文件，或只删文件而遗留可访问记录。
- 不允许把“撤销分享”实现成仅隐藏前台入口但实际链接仍可访问。

### T9 集成验证、样例数据与部署说明

目标：

- 验证主链路可用性。
- 准备最小样例数据。
- 编写本地启动和部署说明。
- 至少验证文件上传、HTML 保存、列表、搜索、详情、分享访问、删除、撤销分享这八条主流程。
- 说明中必须覆盖 PocketBase、业务壳和网页层三者的启动关系。

输入依赖：

- T7
- T8

输出结果：

- 验证记录。
- 启动说明。
- MVP 集成结果。

说明：

- 不允许只验证单接口通不通就宣布 MVP 可用。
- 不允许只写开发者口头说明而缺少可重复执行的启动步骤。

## 3. 串行 / 并行关系

- 必须串行：`T1 -> T2 -> T3 -> T4`
- 可并行：`T5` 与 `T6` 可在 `T4` 完成后并行推进
- 可并行：`T7` 与 `T8` 可在其各自前置完成后交错推进
- 必须串行收尾：`T9` 只能在前台和管理相关核心能力完成后执行

## 4. 风险 task 标记

- `T4`：统一内容写入同时涉及文件系统、数据模型和一致性控制。
- `T5`：分享与 hash 访问直接决定外部访问边界。

## 5. 不允许进入实现期临时假设的说明

- 不允许在实现时再决定是否使用统一内容模型。
- 不允许让前台直接读取 PocketBase 作为临时捷径。
- 不允许先暴露原始文件名 URL，后续再补 hash 路由。
- 不允许把 API Key 验证逻辑散落到各个 handler 中。
- 不允许把 HTML 富文本与文件上传拆成两个完全不同的领域模型。
- 不允许因为实现成本考虑而绕开 `PocketBase`，临时手搓数据库、文件索引和后台底座替代方案。
- 不允许把“参考开源项目”误执行成“替换主底座”。

## 6. 开源复用执行约束

- `PocketBase` 是必须落地的底座，不是可选建议。
- `Gokapi`、`chibisafe`、`filebrowser`、`Pingvin Share` 只提供局部设计参考，不能成为实现期偷偷替换进来的主系统。
- 若某个 task 的实现方案不再以 `PocketBase` 为基础，应判定为越界，不得继续实现。
- review 时必须检查每个 task 是否仍符合“底座固定、业务壳补规则、网页层做访问”的边界。

## 7. 建议的开发顺序摘要

1. 先把底座、业务壳骨架和身份映射打稳。
2. 再完成统一写入能力。
3. 然后分别补查询能力与分享访问能力。
4. 最后接网页层、删除撤销与集成验证。
